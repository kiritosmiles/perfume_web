## 1. 项目概述

### 1.1 项目定位

**情绪人格×香水 AI Agent** 是一款面向 C 端用户的智能香氛推荐系统。系统以大语言模型（LLM）为核心推理引擎，结合 GraphRAG 图结构推理、情感分析和三层分层记忆架构，将用户的情绪状态、人格特征、使用场景与香调知识图谱进行匹配，生成个性化香氛配方与文字解读，实现"情绪驱动型香水推荐"。

系统具备**会话意图感知**能力——自动识别用户当前是"为自己挑选"还是"为他人送礼"，并据此切换不同的上下文数据加载策略和推荐逻辑。

当前以 **Web 部署**为主，架构预留 App 扩展能力。

### 1.2 项目背景

近几年"情绪价值"已从互联网热词演变为真实的消费驱动力。香水品类正在成为"情绪消费"中最典型、增长最快的品类之一。送礼是香水消费的核心场景——大量购买发生在节日、纪念日、生日等情境中。

### 1.3 C 端核心设计原则

| 原则 | 说明 | 设计体现 |
|------|------|---------|
| **低门槛** | 用户不需要懂香调术语 | 游客模式、情绪卡片、场景化引导 |
| **低延迟** | 首字优先 | 全链路并行化、预加载、骨架流式 |
| **低摩擦** | 减少操作负担，隐式优于显式 | 隐式反馈、选项化反问、异步尾量 |
| **被理解感** | 让用户感到"被听见" | 情绪确认微交互、精炼对话、主动回访 |
| **可掌控** | 用户知道 AI 记住了什么 | 记忆透明化、安全档案维护、意图手动切换 |
| **安全感** | 不滥用情绪数据；危机正确响应 | 本地敏感词、危机差异化、数据删除权 |
| **场景智能** | 感知"为谁挑选" | 自用/送礼/探索三种意图差异化策略 |
| **可回溯** | 用户可回顾自己的情绪旅程 | 全量会话持久化、历史浏览、情绪时间线 |

### 1.8 LLM 调用策略总览 🔧 v2.6

系统采用**分层 + 双路径 + 约束化 Agent Gate**策略——输入层 BERT 快路径 / LLM 兜底、推理层按需深浅、精炼层规则+语义 Gate、每层有硬边界。

```

    ┌───────────────────────┴───────────────────────┐
    │  输入处理层 (始终执行)                           │

    │  └────────────────────────────────────────────┘ │
    │                                                 │
    │  ┌─ 文本通道(33%): BERT 主路径(70%) + LLM兜底   │ │
    │  │  ├─ BERT 情绪分类: ~50ms                    │ │
    │  │  │   置信度≥0.7 → 直接使用,跳过LLM           │ │
    │  │  └─ LLM 兜底(30%): 通感解码+意图,~800ms     │ │
    │  └────────────────────────────────────────────┘ │
    │                                                 │
```

### 1.9 LLM 调用约束矩阵 🆕 v2.6

**设计原则**: 每次 LLM 调用必须有硬边界——超时即降级，不等待；最多 1 次重试；耗尽后不兜圈。

| # | 调用点 | 类型 | 正常延迟 | 超时 | 重试 | 重试策略 | 耗尽后行为 | 覆盖场景 |
|---|--------|------|:---:|:---:|:---:|---------|-----------|------|
| ① | BERT 情绪分类 | 本地推理 | 50ms | 200ms | 0 | — | 走 ② LLM 兜底 | 100% 文本对话 |
| ② | LLM 情绪兜底(含通感+意图) | 云端 LLM | 800ms | 1.5s | 1 | 换轻量模型(500ms) | 用 BERT 结果(即使低置信) | ~10% 文本对话 |
| ③ | 调用 B 安全旁路 | 云端 LLM(异步) | 不阻塞 | 2s | 1 | 换轻量模型 | 降级纯本地敏感词,标记 `safety_degraded` | 100% 对话 |
| ④ | 快速模式 LLM 生成 | 云端 LLM | 700ms | 2s | 1 | 换轻量模型(500ms) | 配方骨架缓存兜底(预计算) | ~88% 场景 |
| ⑤ | Deep: Composer | 云端 LLM | 700ms | 1.5s | 1 | 换轻量模型 | Graph 纯检索结果作为最终推荐 | ~12% 场景 |
| ⑥ | Deep: Safety Reviewer | 云端 LLM(轻量) | 400ms | 800ms | 0 | — | 跳过审查，标记 `unreviewed` | ~12% 场景 |
| ⑦ | Deep: Aggregator | 云端 LLM(轻量) | 600ms | 1s | 1 | 换轻量模型 | 规则排序(图谱分数+安全分数+多样性加权) | ~12% 场景 |
| 🆕⑧ | Agent Gate: 信息完整性 | 云端 LLM(轻量) | 200ms | 500ms | 1 | 换轻量模型(200ms) | 当作 `sufficient`，用已有信息生成 | ~5% 场景触发 |
| 🆕⑨ | Agent Gate: 精炼语义 | 云端 LLM(轻量) | 200ms | 500ms | 0 | — | 继续规则精炼（不追问） | ~15% 精炼对话 |

**重试与超时细则**:

| 约束维度 | 规范 |
|---------|------|
| **超时判定** | 从 LLM 请求发出开始计时。到达 timeout 毫秒后无响应 → 视为超时，不等结果返回 |
| **重试间隔** | 0ms（立即重试）。理由：超时已说明当前模型不可达，等待无益 |
| **重试模型切换** | 主模型超时 → 切换到备用轻量模型（如从 Sonnet 4.6 → Haiku 4.5 或 Qwen-7B 微调版）。轻量模型 token 限制减半 |
| **0 次重试的调用** | ①⑥⑨：① 是本地推理不涉及云端；⑥ 是非主链路的尾量审查，跳过不影响安全基线；⑨ 语义不满判断失败时保守行为是"继续精炼"而非追问 |
| **重试也失败** | 执行"耗尽后行为"列中的 degrade 策略。降级输出必须带标记（如 `quality_degraded=true`）供监控 |
| **全链路累计** | 单次用户请求的 LLM 调用总时长（含重试）上限为 **10s**。超过 → SSE 中断并输出"抱歉，我正在经历高负载，请稍后重试"。🔧 v2.7: 从 8s 上调至 10s（深度模式最坏情况: ②2.0s+⑤2.0s+⑦1.5s+⑧0.7s ≈ 6.2s，加上渲染和网络 RTT 余量）。异步调用③不计入累计 |

**降级标记透传**:

```
每次 LLM 调用的结果对象:
{
  "result": {...},
  "metadata": {
    "call_id": "④",
    "latency_ms": 720,
    "retry_used": false,
    "model_used": "sonnet-4-6",
    "degraded": false,
    "fallback_reason": null
  }
}

降级示例:
{
  "result": {...},           ← 规则排序结果(非 LLM)
  "metadata": {
    "call_id": "⑦",
    "latency_ms": 1050,
    "retry_used": true,
    "retry_model": "haiku-4-5",
    "degraded": true,
    "fallback_reason": "timeout_after_retry"
  }
}
```

前端根据 `degraded` 标记在对应内容区域展示淡色提示（如「基于已有信息推荐」），但不阻断用户继续交互。

```
              │
  150-650     🆕⑧ Agent Gate: 信息检查              budget=500ms, retry=1
              │ LLM 判断: insufficient
              │   missing_dims: [收礼人身份, 偏好, 过敏史]
              │   核心维度缺失，工具无法补充
              │ ⭐ 整理追问清单:
              │   ① 妈妈平时工作是什么类型？

              │   ③ 她的皮肤对某些气味敏感吗？
              │
  650         ⭐ 首响应: 追问输出 (非推荐)
              │ LLM 调用: 1 次(Gate) ✅
              │ 延迟: ~0.65s
              │
              ▼
     [用户回答追问后，重新进入 Gate → sufficient → 生成推荐]
```

## 3. 业务模块二：情绪识别与分析

### 3.1 模块概述

多维输入 → 情绪分布向量 + 场景向量 + 环境向量融合。

### 3.2 功能需求

#### FR-2.1 双通道情绪输入

- 通道 A: 自由文本。通道 B: 情绪卡片（多选，最多 2 张，预设向量）。

#### FR-2.2 情绪识别 + 意图 + 通感（调用 A）🔧 v2.5.1

- **需求描述**: 采用**双路径分级**策略——高频路径走 BERT 轻量分类（~50ms），低频路径走 LLM 兜底（~800ms）。意图分类始终走关键词优先（< 1ms）+ LLM 兜底。
- **🆕 v2.5.1 双路径架构**:

```
用户文本输入
    │
    ├──→ 路径A: BERT 情绪分类 (主路径,覆盖 ~70% 对话)
    │     模型: bge-emotion / 自训练多标签分类器
    │     延迟: ~50ms (GPU推理) / ~100ms (CPU推理)
    │     输出: 情绪分布向量 {"焦虑": 0.6, "平静": 0.2, ...}
    │
    ├──→ 路径B: LLM 通感+意图兜底 (仅 ~30% 对话)
    │     触发: BERT 置信度 < 0.7 或 检测到通感表达 或 送礼关键词
    │     延迟: ~800ms (热启动)
    │     输出: 完整结构化输出 (情绪修正 + 意图 + 通感)
    │
    └──→ 关键词意图检测 (始终执行,< 1ms)
          送礼关键词命中 → 直接标记 gift
```

- **BERT 模型选型**: `bge-large-emotion` 或基于中文情绪标注数据微调的 `bert-base-chinese`。输出 8 类情绪标签 + 强度（对应情绪卡片的 8 个类别）。
- **🔧 v2.7 微调数据策略**:
  - 初始微调数据来源: 上线后用 LLM（调用 A）对生产对话进行情绪标注（人工抽样校验 ≥ 200 条/月），逐步积累领域标注数据
  - MVP Phase 1 策略: 优先使用零样本 `bge-large-emotion` 作为基线。5K 条人工标注数据对 8 类情绪标签微调存在过拟合风险（每类 ~600 条），应在积累 ≥ 2 万条领域标注数据后再做微调
  - 标注质量控制: LLM 标注结果由 2 名标注员交叉校验（Cohen's Kappa ≥ 0.7），分歧样本由第三人仲裁
  - 来源: v2.7 评审修复 (#30)
- **路径选择逻辑**:
  - BERT 置信度 ≥ 0.7 且无通感关键词 → 纯 BERT 结果，跳过 LLM。
  - BERT 置信度 < 0.7 或检测到通感模式 → 启动 LLM 路径 B，以 BERT 结果作为 few-shot 锚点。
  - 🔧 **v2.7 通感触发精化**: 通感关键词（"像""味道""感觉"）必须与抽象名词共现才触发。匹配模式: `{像|好像|仿佛} + {一种|某种} + {抽象名词}` 或 `{抽象名词} + {的} + {味道|感觉|气息}`。单独出现"我感觉这个不错"不触发通感路径。
  - 情绪卡片通道（FR-2.1 通道 B）完全绕过两条路径——预设向量直接注入。
- **延迟收益**: 70% 的文本对话从 LLM 1.5s 降至 BERT 50ms + 意图 1ms = **~50ms**，输入处理层延迟降低 ~97%。
- **来源**: v2.4 评审修复 (#6)，v2.5.1 云端延迟校准

#### FR-3.7 推荐精炼对话 🔧 v2.6

- **需求描述**: 用户对推荐结果提出调整需求时，系统以最低延迟完成局部调整。精炼不走完整推理链路，只在当前候选集上做局部操作。同时引入语义 Agent Gate 防止盲目录调整。
- **精炼流程（v2.6 修订）**:

```
Turn N: 用户精炼请求
    │
    ├── 第 1 次精炼: 规则引擎直接处理 (< 500ms)
    │   规则覆盖维度（🔧 v2.7 穷举）:
    │   ┌──────────┬──────────────────────────────────────┐
    │   │ 维度     │ 可处理指令示例                        │
    │   ├──────────┼──────────────────────────────────────┤
    │   │ 甜度     │ "太甜"/"不够甜"/"更甜一点"           │
    │   │ 浓度     │ "太浓"/"太淡"/"更清淡"               │
    │   │ 清新度   │ "更清新"/"太沉闷"                    │
    │   │ 性别倾向 │ "更女性化"/"更男性化"/"太中性"       │
    │   │ 温暖度   │ "太冷"/"更温暖"                      │
    │   │ 香调切换 │ "换个香调"/"试试花香"                │
    │   │ 场合切换 │ "更正式"/"更休闲"                     │
    │   │ 年龄适配 │ "太年轻"/"太成熟"                    │
    │   │ 季节适配 │ "更适合夏天用"                        │
    │   └──────────┴──────────────────────────────────────┘
    │   不可处理: "太贵了"(价格,MVP范围外) / "前调太冲"(三调精细调整)
    │   不可处理 → 走第2次精炼 Agent Gate 语义判断 → 追问或升级
    │   操作: 在候选集中调整对应维度的权重，无需 LLM
    │
    ├── 🆕 第 2 次精炼 (v2.6):
    │   ├── 先尝试规则引擎解析
    │   └── Agent Gate: 语义判断 (轻量 LLM, 200ms, 0 重试):
    │       ├── 规则可处理 → 继续规则精炼 (< 500ms)
    │       ├── 语义不满("没理解我" "还是不对" "不是这个意思")
    │       │   → ⭐ 停止精炼，降级到追问
    │       │   → 输出: "让我换个思路——
    │       │       你刚才说的[情绪/场景]，我理解对了吗？
    │       │       可以告诉我更多关于[缺失维度]吗？"
    │       └── 超时(500ms) → 继续规则精炼（保守行为）
    │
    └── 第 3 次精炼:
        ├── 规则引擎再次尝试
        └── 依然解析失败 → 自动升级深度模式 (FR-3.12)
            文案：「你已经调整了 3 次——让我换个思路，从不同的角度重新为你搭配 🌿」
            深度模式产生的推荐作为全新推荐起点，精炼计数器重置
```

- **关键设计**: 语义 Agent Gate 仅在精炼第 2 次时触发——避免第 1 次就过度反应（可能只是常规调整），也避免第 3 次才意识到用户不满（已经浪费了 2 轮）。0 次重试确保保守行为：判断失败 → 继续精炼，不引入额外阻塞。
- **延迟预算**: 第 1 次 < 500ms（纯规则）。第 2 次 < 500ms（规则可处理）或 ~200ms（Agent Gate→追问，远快于继续精炼并让用户不满）。
- 🆕 v2.4 精炼对话前的意图再确认: 如果上一轮和当前轮的 LLM 意图判定不一致，**立即**触发轻量确认——不等到第 2 次精炼的 Agent Gate。意图变化优先级高于精炼语义判断：先确认意图，再执行精炼操作。
  - 🔧 v2.7: 明确意图变化时跳过正常精炼流程。意图变化检测在所有精炼操作前执行（包括第 1 次）。确认后重置精炼计数器。
  - 示例: 用户在为自己精炼时说"太甜了，她应该也不喜欢" → 检测到 gift 关键词 → 立即确认"你是想为自己调整还是为她调整？"
- 精炼退出机制：深度模式后如仍不满意 → 收藏+稍后继续模式，展示所有已尝试版本。
- **来源**: v2.4 评审修复 (#11)，v2.5 架构升级，🆕 v2.6 Agent Gate 语义理解

#### FR-3.12 深度模式 — Supervisor+Subagent 推理 🔧 v2.5.1

- **需求描述**: 在高价值推荐场景下，系统自动从快速模式升级为深度模式——由 Supervisor 分析上下文并制定多角度探索策略，调度 3 个独立 Subagent 并行推理，Aggregator 汇聚排序后输出最优推荐。
- **设计原则**: 深度模式**不是默认路径**——约 88% 的日常推荐走快速模式（单次 LLM），约 12% 的高价值场景触发深度模式（🆕 v2.6 修正：部分原升级场景被 Agent Gate 追问替代）。
- **触发条件（任一满足即触发）**:

| # | 触发条件 | 场景 | 说明 |
|---|---------|------|------|
| 1 | **精炼 3 轮后仍不满意** | 用户连续 3 次调整后仍有修改意愿 | FR-3.7 自动升级 |

- **冷归档查询策略**: Layer 1+2 热检索未找到足够上下文时（topK 的相似度均 < 阈值），触发冷归档层查询。用户通常不需要等待冷归档结果——冷归档**异步追加上下文**，在后续精炼轮次中注入。

### 5.3 功能需求

#### FR-4.2 Layer 1 — Agent 会话记忆 🔧 v2.4

- **容量**: 1 轮首次对话（全量 Q&A）+ 5 轮最近对话（全量 Q&A）= 最多 6 轮。
- **🔧 v2.7 压缩规则（数学已修正 + 压缩比上限）**: 
  - 触发条件：第 7 轮对话开始时（总轮次 > 6）。
  - 压缩对象：第 2 轮至第 (N-1) 轮（即**不含首次**、**不含最后一轮**的所有中间轮次）。
  - 压缩输出：
    - 中间轮次 ≤ 6 轮 → 1 条结构化摘要
    - 🆕 中间轮次 ≥ 7 轮 → 分批压缩为 **2 条摘要**（每条约 3-4 轮），保证每条摘要的信息密度合理
    - 写入 Layer 2 热检索层
  - 压缩后 Layer 1 保留：首次对话（全量）+ 最后一轮对话（全量）= 2 轮。
  - 后续每次新轮次到来：继续累加，每次压缩将当前的中间轮次压缩为 1~2 条摘要，Layer 1 始终回到"首次 + 最后一轮"的 2 轮状态。
  - 🆕 **压缩比上限**: 单条摘要的输入上限为 6 轮对话。超过 6 轮中间对话时自动分批。防止 10+ 轮对话压缩为 1 条摘要导致信息严重丢失。
- **来源**: v2.4 评审修复 (#5)，v2.7 评审修复 (#33)

  - `high` 危机转接：24 小时内人工回复

#### FR-5.6 危机场景差异化交互 🔧 v2.7

- **需求描述**: 当检测到用户处于心理危机状态时，系统立即切换交互模式——暂停推荐、避免轻浮语气、引导支持资源。
- 🔧 **v2.7 危机检测机制**:

```
用户消息
    │
    ├── 第一层: 危机关键词字典 (本地, < 1ms)
    │   字典包含: ["自杀","不想活了","结束生命","自残","想死"...
    │            "活不下去了","没有意义了","想消失"...]
    │   命中 → 🔴 直接标记为"疑似危机"，无需二三层
    │
    ├── 第二层: BERT/LLM 语义兜底 (归入调用 B，异步)
    │   关键词未命中但文本长度 > 20 字 → 调用 B 增加 crisis 判定维度
    │   输出: {crisis_detected: boolean, severity: "low"|"medium"|"high"|"none"}
    │   判定为 medium/high → 🔴 触发危机响应
    │
    └── 备注: 调用 B 可能因超时降级(§1.9 ③) →
       如果调用 B 不可用，危机关键词字典作为 P0 底线保障（不依赖 LLM）
```

- **检测性能要求**:
  - 关键词命中 → **不可漏判（P0）**：宁可误判不可漏判。关键词字典由心理健康专业人员审核，按月维护
  - LLM 语义 → 目标召回 ≥ 95%（假阴性率 < 5%），误报容忍度较高（< 20% 假阳性可接受——误触发"暂停推荐"优于漏判）
  - 假阳性处理：如果触发危机响应但用户表示"我没事，只是随口一说" → 记录为误判样本，用于改进 LLM prompt

- 🔧 **v2.7 危机响应流程**:

```
危机检测触发 (关键词 或 LLM medium/high)
    │
    ├── ⭐ 立即暂停当前推荐生成（SSE 中断）
    ├── 开场白切换为关怀语气（禁止推荐任何香调）
    │   "我注意到你提到了一些让我担心的内容。
    │    作为 AI，我无法提供专业心理支持，但希望你知道——你不需要一个人面对这些。"
    │
    ├── 展示可配置的心理援助热线（按地区匹配）
    │   格式: FR-5.6 的 JSON 配置
    │
    ├── 后续交互限制:
    │   ├── 本轮 session 禁止启动任何推荐（快速/深度/精炼均禁用）
    │   ├── 用户仍可自由对话（不被"禁言"），但系统不再主动提供推荐
    │   └── 用户手动切换意图标签或新 session 后恢复正常
    │
    └── 安全日志: 记录 crisis 事件（detection_method, severity, timestamp）
        独立于普通审计日志，限权访问
```

- **危机 vs 普通负面情绪区分**:
  
| 维度 | 普通负面情绪 | 危机信号 |
|------|-----------|---------|
| 关键词 | "焦虑""难过""压力大" | "不想活了""结束一切""没有希望" |
| 语境 | 有具体触发事件（工作/感情/学业） | 无具体事件，或事件 + 自我否定 |
| 时间维度 | 临时状态（"今天""最近"） | 持续/永久（"一直""永远"） |
| 行为意向 | 寻求帮助/建议 | 表达无望/结束意图 |
| 示例 | "今天工作压力好大" | "我觉得活着好累，没有意义了" |

- 🔧 v2.4 **心理援助热线可配置化**: （保持，同 v2.4）

#### FR-5.7 会话意图检测 + 信息完整性判定 🆕 v2.6

- **需求描述**: 检测用户当前会话意图（self_use / gift / explore），并同步输出信息完整性维度，供 Agent Gate（FR-5.11）使用。
- **判定流程（v2.6 修订）**:

```
用户消息 + 对话历史
    │
    ├── 第一层: 关键词检测 (< 1ms)
    │   ├─ "给自己买" "我想要的" "自用" → self_use
    │   ├─ "送给" "给...买" "礼物" "妈妈/女朋友/朋友" → gift
    │   └─ 不确定 → 进入第二层
    │
    ├── 第二层: LLM 调用 A 合并判定 (含意图+情绪+通感, ~800ms)
    │   输出: {
    │     intent: "self_use" | "gift" | "explore",
    │     intent_confidence: 0.0~1.0,
    │     🆕 info_completeness: {          ← v2.6 新增
    │       level: "sufficient" | "partial" | "insufficient",
    │       missing_dims: ["收礼人身份", "收礼人偏好", ...],
    │       can_fill_from_tools: boolean,
    │       suggested_tool_call: string | null
    │     }
    │   }
    │
    └── 第三层: 用户显式切换
        前端始终展示意图标签 (🌿为自己 / 🎁为他人 / 🔍随便看看)
        用户可随时点击切换，手动切换直接覆盖 LLM 判定
```

- **信息完整性维度**（🆕 v2.6）:

| 意图 | 核心必需维度 | 缺失时可工具补的维度 | 必须追问的缺失 |
|------|------------|-------------------|-------------|
| `self_use` | 情绪+场景 | 偏好历史(Layer2)、安全档案(Layer3) | 场景完全未知 |
| `gift` | 收礼人身份+场景 | 收礼人偏好(送礼历史) | 收礼人完全未知(没有任何标签) |
| `explore` | 探索方向 | 几乎所有维度都可缺省 | 无——探索模式本就允许信息稀疏 |

- `info_completeness` 输出直接传递给 Agent Gate（FR-5.11），Gate 根据 `level` 决定跳过/补工具/追问。
- **来源**: v2.2 意图感知，🆕 v2.6 信息完整性维度

#### FR-5.8 抽象/超现实需求处理 🔧 v2.4

- LLM 通感解码 → 香调转译 → 确认微交互。
- 🔧 v2.4 **解码后处理修正**: 通感解码后的香调词汇**不替代**情绪向量，而是作为 GraphRAG 检索的**引导种子节点**。即直接在知识图谱中从解码香调出发做子图扩散，跳过情绪→香调的通用映射环节。这避免了在向量融合层面对不同语义空间的向量做不当替换。
- 示例: "下雨的味道" → 解码为 [臭氧调, 湿润木质, 青草] → 以这三个节点为种子在 Neo4j 中扩散 2 跳 → 收集关联香调和原料 → 融入推理。
- **来源**: v2.4 评审修复 (#8)

#### FR-5.11 信息完整性 Agent Gate 🆕 v2.6

- **需求描述**: 在推理生成前插入一个有硬边界的决策节点——判断当前上下文是否足够支撑高质量推荐，不够则限时收集信息或整理追问。
- **定位**: 上下文组装完成后、推理生成开始前。不阻塞主流程中的已有并行操作（BERT、记忆检索、GraphRAG）。
- **Gate 判断逻辑**（轻量 LLM, budget 500ms, retry=1）:

```
上下文组装完成
    │  输入: 用户消息 + 意图 + 情绪 + 场景标签 + 记忆上下文
    │        + GraphRAG 候选数 + FR-5.7 的 info_completeness
    │
    ▼
┌─────────────────────────────────┐
│  Agent Gate: 信息完整性检查      │
│  预算: 500ms + 1 次重试          │
└─────────────┬───────────────────┘
              │
  ┌───────────┼───────────┐
  ▼           ▼           ▼
✅ 信息充足  🟡 可补充   🔴 信息不足
(sufficient) (partial)  (insufficient)
  │           │           │
  ▼           ▼           ▼
跳过 Gate    调 1 次工具  整理追问清单
0ms 开销     超时 200ms   输出给用户
  │           │           │
  │    ┌──────┴──────┐    │
  │    ▼             ▼    │
  │  查到补充    超时/未果 │
  │  合并context  仍不足   │
  │    │             │     │
  ▼    ▼             ▼     ▼
  进入推理生成      进入推理生成   输出追问 🌿
                   (标记partial)
```

- **判定规则表**:

| 条件 | 判定 | 行为 | 示例 |
|------|:---:|------|------|
| 所有核心维度齐备 + GraphRAG 候选 ≥ 5 | `sufficient` | 跳过 Gate，0ms 开销 | 老用户自用推荐，情绪/场景/偏好齐备 |
| 1-2 个维度缺失，但工具可补充 | `partial` | 调 1 次工具(≤200ms)，补上 → 生成 | 送礼场景，收礼人偏好缺失但送礼历史存在 |
| 核心维度缺失且工具无法补充 | `insufficient` | 整理追问清单，直接输出给用户 | 送礼场景，收礼人完全未知且无历史 |
| LLM 超时(500ms) | — | 当作 `sufficient`，用已有信息生成 | 网络抖动导致 Gate LLM 超时 |
| 重试也超时 | — | 当作 `sufficient`，标记 `gate_bypassed` | — |

- **与意图的联动**:

| 意图 | sufficient 条件 | insufficient 触发 | 追问方向 |
|------|:---:|------|------|
| `self_use` | 情绪已知 + 场景已知 + 安全档案已加载 | 场景标签完全未知 | "你是想要日常用的，还是特定场合？" |
| `gift` | 收礼人身份标签 ≥ 2 项(如年龄+关系) + 场景 | 收礼人完全未知(0 标签) | "先了解几个小细节 🌿 ① 收礼人年龄范围？② 平时工作/生活类型？③ 以前用过什么香水？" |
| `explore` | 探索方向已知 | **几乎不触发** | 探索模式本身允许信息稀疏 |

- **关键设计决策**:
  - **不是每轮都调用**: Gate 在 95% 场景下判定 `sufficient` → 0ms 开销。仅在 ~5% 场景触发 LLM。
  - **追问优于猜测**: 信息不足时系统不说"我猜你想要..."，而是明确追问。追问文案由 Agent Gate 输出 `suggested_questions` 格式化后展示。
  - 🔧 **v2.7 追问质量保障**:
    - 追问数量上限 ≤ 3 个（避免问题轰炸）
    - 追问不能涉及 PII（"你妈妈叫什么名字？""她的手机号？"等）——由轻量正则 + 敏感词库过滤
    - 追问在展示前经本地敏感词过滤（jieba）
    - 追问应自然、非侵入式：避免"请提供以下信息"的表格化语气，改为对话式引导
  - **硬边界**: 500ms 超时 + 1 次重试，耗尽后绝不阻塞——标记 `gate_bypassed` 并继续生成。
  - **探索模式豁免**: `explore` 意图下 Gate 几乎不触发 insufficient，因为探索本身即允许信息稀疏。
- **延迟预算**（§1.9 调用⑧）:
  - sufficient 判定: 0ms（跳过 LLM）
  - partial 判定 + 工具补充成功: +200ms
  - insufficient 判定 → 追问: ~200ms（远快于生成，用户感知为"秒回"）
  - 超时 → bypass: +500ms（最坏情况，但概率极低）
- **监控指标**: Gate 触发率（目标 ~5%）、insufficient→追问率（目标 ~2%）、Gate 超时率、追问后用户回复率（衡量追问质量）。
- **来源**: v2.6 Agent Gate 混合模式设计

| 残余瓶颈 | 当前延迟 | 进一步优化方向 | 预期收益 |

### 14.6 核心结论

1. **BERT 情绪分类是延迟优化的最大杠杆**：将 70% 文本对话的情绪识别从 LLM 1.5s 降至 BERT 50ms，单次对话节省 1.45s，加权平均收益显著。
2. **Supervisor 去 LLM 化是关键架构决策**：触发条件已在 FR-3.12 中明确定义 5 种场景，每种场景的分派策略可通过策略矩阵硬编码，无需 LLM 动态判断。
3. **v2.7 加权平均 ~1.37s 在 C 端可接受**：v2.5.1 基线 ~1.37s + Agent Gate 增量 ~10ms ≈ ~1.37s（🔧 v2.7 百分比修正后持平）。前提是情绪卡片路径占主流（67%）。如果文本路径占比超过 50%，平均将升至 ~1.6s（仍可接受，BERT 保障）。
4. **深度模式 1.7s 骨架是云端模型物理极限**：即使经过全部优化，Supervisor+Subagent×2+Aggregator 的串行结构在云端模型上无法突破 1.5s。如需进一步下降，需要边缘部署（SDK/本地端侧推理）。
5. **精炼对话 < 500ms 完全不受云端影响**：规则引擎 + 局部图谱操作，不涉及 LLM 推理。这是全系统延迟最稳的环节。
6. 🆕 **Agent Gate 的延迟代价极低（加权 ~10ms）**：95% 场景 Gate 判定 sufficient 后直接跳过（0ms）。partial 场景的 +200ms 工具补充改善了推荐质量。insufficient→追问 的首响应 ~0.6s 远快于勉强生成劣质推荐——这是"延迟换质量"的正收益交换。
7. 🆕 **全链路 LLM 调用约束矩阵（§1.9）保障了最坏情况边界**：每层超时+重试+降级策略确保单次 LLM 调用失败不会拖垮整个链路的延迟。全链路累计 LLM 调用上限 10s（🔧 v2.7: 从 8s 上调，深度模式最坏情况 ~6.2s + 余量）。
8. 🆕 **Agent Gate 不是通用 Agent 循环，而是 2 个精确节点**：信息完整性 Gate 仅在推理前触发（~5% 场景触发 LLM），精炼语义 Gate 仅在第 2 次精炼后触发（~15% 精炼对话）。两个节点各 ≤500ms 硬边界。在获得 Agent 灵活性的同时保持确定性编排的延迟可预测性。
