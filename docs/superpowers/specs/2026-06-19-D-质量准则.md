## §1 性能指标

> 详见产品需求文档 §9.1 性能指标（v2.6 更新）

以下 18 项可度量目标覆盖端到端全链路——从首 Token 到全量会话列表加载，从缓存命中到吞吐量与触发率。所有指标均须在 CI/CD 流水线中自动化校验，P99 数据取生产环境连续 7 天观测值。

| # | 指标 | 目标值 | 说明 |
|---|------|--------|------|
| 1 | 对话首 Token 时间（TTFT）— 云端热 | P99 < 500ms | Prompt Cache 命中时 SSE 第一个 token |
| 2 | 对话首 Token 时间（TTFT）— 云端冷 | P99 < 800ms | 冷启动时 |
| 3 | LLM 调用 A（情绪文本通道） | P99 < 800ms | BERT 分类主路径（~50ms）+ LLM 通感兜底（仅 ~30% 对话触发） |
| 4 | LLM 调用 B（安全旁路） | 不阻塞主流程 | 异步，异常时 SSE 中断 |
| 5 | Layer 1 检索 | < 5ms | Redis 内存读取 |
| 6 | Layer 2 热检索（缓存命中） | < 20ms | Redis 语义检索 |
| 7 | Layer 2 热检索（缓存未命中） | < 200ms | 外部存储回源 |
| 8 | Layer 2 冷归档检索 | < 1s | 异步，不阻塞主流程 |
| 9 | Layer 1→2 压缩 | < 3s（后台异步） | — |
| 10 | 快速模式骨架（情绪卡片） | < 1.5s | 约占 67% 对话，95% 场景 Agent Gate 跳过(0ms 增量) |
| 11 | 快速模式骨架（文本） | < 2s | BERT 情绪 + Prompt Cache 预热后可达 |
| 12 | 深度模式骨架（优化后） | < 3s | Supervisor 去 LLM 化 + Subagent 并行 |
| 13 | 深度模式完整输出 | < 4.5s | + Safety Reviewer + Aggregator |
| 14 | Agent Gate partial 补充 | +200ms | 仅 ~3% 场景触发，调 1 次工具补充信息 |
| 15 | Agent Gate insufficient→追问 | ~200ms | 仅 ~2% 场景触发，首响应远快于生成 |
| 16 | Agent Gate 超时→bypass | +500ms | 极低概率，最坏情况退化 |
| 17 | 加权平均骨架延迟（v2.7） | ~1.37s | 百分比修正后 ~1.35s + Gate 增量 ~10ms = ~1.37s |
| 18 | 精炼语义 Agent Gate | +200ms（仅第2次精炼） | ~15% 精炼对话触发；规则可处理时 0ms 增量 |

**补充指标**:

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 精炼对话响应（第1次/规则可处理） | < 500ms | 规则引擎 + 局部图谱，不受云端延迟影响 |
| SSE 断连重连 | < 3s | 骨架级续传 |

## §2 安全策略

> 详见产品需求文档 FR-5.2、FR-5.6、FR-5.9、FR-5.10、§9.4

### 2.1 敏感词过滤

> 详见产品需求文档 FR-5.2

- **实现方式**: 纯本地 jieba + 敏感词库，异步旁路，不阻塞主流程。
- **词库生命周期**:
  - **初始来源**: 公开敏感词库 + 手工审核 → 基线词库。
  - **维护周期**: 运营人员按月维护。
  - **补充机制**: 用户举报作为补充来源，经人工审核后入库。
  - **审计**: 所有词库变更记录不可篡改的审计日志（谁、何时、增删了什么、原因）。

### 2.2 危机检测质量

> 详见产品需求文档 FR-5.6

**三层检测架构**:

```
用户消息
    │
    ├── 第一层: 危机关键词字典 (本地, < 1ms)
    │   字典包含: ["自杀","不想活了","结束生命","自残","想死"...
    │            "活不下去了","没有意义了","想消失"...]
    │   命中 → 🔴 直接标记为"疑似危机"，无需二三层
    │
    ├── 第二层: BERT/LLM 语义兜底 (归入调用 B，异步)
    │   关键词未命中但文本长度 > 20 字 → 调用 B 增加 crisis 判定维度
    │   输出: {crisis_detected: boolean, severity: "low"|"medium"|"high"|"none"}
    │   判定为 medium/high → 🔴 触发危机响应
    │
    └── 备注: 调用 B 可能因超时降级(§1.9 ③) →
       如果调用 B 不可用，危机关键词字典作为 P0 底线保障（不依赖 LLM）
```

**检测性能要求**:

| 要求 | 规范 |
|------|------|
| P0 原则 | 关键词命中 → **不可漏判（P0）**：宁可误判不可漏判。关键词字典由心理健康专业人员审核，按月维护 |
| LLM 语义召回率 | 目标召回 >= 95%（假阴性率 < 5%） |
| 假阳性容忍度 | < 20% 假阳性可接受——误触发"暂停推荐"优于漏判 |
| 假阳性处理 | 如果触发危机响应但用户表示"我没事，只是随口一说" → 记录为误判样本，用于改进 LLM prompt |

**危机 vs 普通负面情绪区分**:

| 维度 | 普通负面情绪 | 危机信号 |
|------|-----------|---------|
| 关键词 | "焦虑""难过""压力大" | "不想活了""结束一切""没有希望" |
| 语境 | 有具体触发事件（工作/感情/学业） | 无具体事件，或事件 + 自我否定 |
| 时间维度 | 临时状态（"今天""最近"） | 持续/永久（"一直""永远"） |
| 行为意向 | 寻求帮助/建议 | 表达无望/结束意图 |
| 示例 | "今天工作压力好大" | "我觉得活着好累，没有意义了" |

**危机响应流程**:

```
危机检测触发 (关键词 或 LLM medium/high)
    │
    ├── ⭐ 立即暂停当前推荐生成（SSE 中断）
    ├── 开场白切换为关怀语气（禁止推荐任何香调）
    ├── 展示可配置的心理援助热线（按地区匹配）
    ├── 后续交互限制:
    │   ├── 本轮 session 禁止启动任何推荐
    │   ├── 用户仍可自由对话
    │   └── 用户手动切换意图标签或新 session 后恢复正常
    └── 安全日志: 记录 crisis 事件（detection_method, severity, timestamp）
        独立于普通审计日志，限权访问
```

### 2.3 Agent 角色边界保护

> 详见产品需求文档 FR-5.9

- **三级检测**: `normal` / `borderline` / `overstep`
- **实现**: 检测移至 LLM 调用 B（安全旁路），不再阻塞主流程。命中 `overstep` 时通过 SSE 中断机制介入。
- **升级规则**: 连续 3 次 `overstep` → **强制建议联系人工**（与 FR-5.3 转人工机制联动）。

### 2.4 越狱/辱骂防护

> 详见产品需求文档 FR-5.10

**四层防护体系**:

| 层级 | 机制 | 说明 |
|:---:|------|------|
| 1 | 敏感词过滤 | 本地 jieba 异步旁路（与 FR-5.2 共用词典） |
| 2 | LLM 语义检测 | 调用 B 安全旁路，输出 `safe` / `hostile` / `injection_attempt` |
| 3 | 响应消解 | 不重复、不转述、不执行注入内容 |
| 4 | 系统限制 | SSE 中断 + 安全日志 |

**Prompt Injection 防护**: 检测到后 **不重复、不转述、不执行** 注入内容。

### 2.5 KMS + 加密 + 审计

> 详见产品需求文档 §9.4

| 要求 | 说明 |
|------|------|
| 用户数据加密 | TLS + AES-256 |
| 密钥管理 (KMS) | 加密密钥使用云厂商 KMS 或 HashiCorp Vault 管理。密钥按季度轮换。密钥访问记录审计日志 |
| 情绪数据保护 | 敏感信息 + 危机场景独立权限 |
| 用户授权机制 | 明确告知数据采集范围（含全量会话存储、冷归档永久保留），获取显式同意 |
| 数据删除权 | 全量数据删除（含冷归档层）；30 天软删除 + 物理删除 |
| 审计日志 | 全操作 + 偏好变更 + 意图切换 + 摘要写入/淘汰 + 敏感词库变更 + KMS 访问 |
| 第三方依赖最小化 | 本地敏感词 + 城市级天气 |

## §3 LLM 可靠性

> 详见产品需求文档 §1.9 LLM 调用约束矩阵

**设计原则**: 每次 LLM 调用必须有硬边界——超时即降级，不等待；最多 1 次重试；耗尽后不兜圈。

### 3.1 LLM 调用约束矩阵

| # | 调用点 | 类型 | 正常延迟 | 超时 | 重试 | 重试策略 | 耗尽后行为 | 覆盖场景 |
|---|--------|------|:---:|:---:|:---:|---------|-----------|------|
| ① | BERT 情绪分类 | 本地推理 | 50ms | 200ms | 0 | — | 走 ② LLM 兜底 | 100% 文本对话 |
| ② | LLM 情绪兜底(含通感+意图) | 云端 LLM | 800ms | 1.5s | 1 | 换轻量模型(500ms) | 用 BERT 结果(即使低置信) | ~10% 文本对话 |
| ③ | 调用 B 安全旁路 | 云端 LLM(异步) | 不阻塞 | 2s | 1 | 换轻量模型 | 降级纯本地敏感词,标记 `safety_degraded` | 100% 对话 |
| ④ | 快速模式 LLM 生成 | 云端 LLM | 700ms | 2s | 1 | 换轻量模型(500ms) | 配方骨架缓存兜底(预计算) | ~88% 场景 |
| ⑤ | Deep: Composer | 云端 LLM | 700ms | 1.5s | 1 | 换轻量模型 | Graph 纯检索结果作为最终推荐 | ~12% 场景 |
| ⑥ | Deep: Safety Reviewer | 云端 LLM(轻量) | 400ms | 800ms | 0 | — | 跳过审查，标记 `unreviewed` | ~12% 场景 |
| ⑦ | Deep: Aggregator | 云端 LLM(轻量) | 600ms | 1s | 1 | 换轻量模型 | 规则排序(图谱分数+安全分数+多样性加权) | ~12% 场景 |
| ⑧ | Agent Gate: 信息完整性 | 云端 LLM(轻量) | 200ms | 500ms | 1 | 换轻量模型(200ms) | 当作 `sufficient`，用已有信息生成 | ~5% 场景触发 |
| ⑨ | Agent Gate: 精炼语义 | 云端 LLM(轻量) | 200ms | 500ms | 0 | — | 继续规则精炼（不追问） | ~15% 精炼对话 |

### 3.2 重试与超时细则

| 约束维度 | 规范 |
|---------|------|
| **超时判定** | 从 LLM 请求发出开始计时。到达 timeout 毫秒后无响应 → 视为超时，不等结果返回 |
| **重试间隔** | 0ms（立即重试）。理由：超时已说明当前模型不可达，等待无益 |
| **重试模型切换** | 主模型超时 → 切换到备用轻量模型（如从 Sonnet 4.6 → Haiku 4.5 或 Qwen-7B 微调版）。轻量模型 token 限制减半 |
| **0 次重试的调用** | ①⑥⑨：① 是本地推理不涉及云端；⑥ 是非主链路的尾量审查，跳过不影响安全基线；⑨ 语义不满判断失败时保守行为是"继续精炼"而非追问 |
| **重试也失败** | 执行"耗尽后行为"列中的 degrade 策略。降级输出必须带标记（如 `quality_degraded=true`）供监控 |
| **全链路累计** | 单次用户请求的 LLM 调用总时长（含重试）上限为 **10s**。超过 → SSE 中断并输出"抱歉，我正在经历高负载，请稍后重试"。异步调用③不计入累计 |

### 3.3 降级标记透传

每次 LLM 调用的结果对象:

```json
{
  "result": {...},
  "metadata": {
    "call_id": "④",
    "latency_ms": 720,
    "retry_used": false,
    "model_used": "sonnet-4-6",
    "degraded": false,
    "fallback_reason": null
  }
}
```

降级示例:

```json
{
  "result": {...},
  "metadata": {
    "call_id": "⑦",
    "latency_ms": 1050,
    "retry_used": true,
    "retry_model": "haiku-4-5",
    "degraded": true,
    "fallback_reason": "timeout_after_retry"
  }
}
```

前端根据 `degraded` 标记在对应内容区域展示淡色提示（如「基于已有信息推荐」），但不阻断用户继续交互。

### 3.4 链式降级示例——信息不足时的完整超时→追问链路

```
时间轴 (ms)    操作                             约束状态
──────────────────────────────────────────────────────────
   0          用户消息: "给妈妈买香水,不太确定她喜欢什么"
   0-50       ① BERT 情绪分类 → 成功(期待 0.72)
  50-150      并行: 意图(gift) + Layer1+2+3 + GraphRAG
  150         上下文组装 → 送礼场景，收礼人标签为空
              │
  150-650     ⑧ Agent Gate: 信息检查              budget=500ms, retry=1
              │ LLM 判断: insufficient
              │   missing_dims: [收礼人身份, 偏好, 过敏史]
              │   核心维度缺失，工具无法补充
              │ ⭐ 整理追问清单:
              │   ① 妈妈平时工作是什么类型？
              │   ② 之前有没有用过她很喜欢的香水？
              │   ③ 她的皮肤对某些气味敏感吗？
              │
  650         ⭐ 首响应: 追问输出 (非推荐)
              │ LLM 调用: 1 次(Gate) ✅
              │ 延迟: ~0.65s
              │
              ▼
     [用户回答追问后，重新进入 Gate → sufficient → 生成推荐]
```

### 3.5 LLM 调用兜底策略

> 详见产品需求文档 FR-7.4

**四层兜底体系**:

```
主模型 (Primary) → 降级 (Degraded,轻量模型) → 缓存兜底 (Cache Fallback) → 规则引擎 (Rule Engine)
```

- LLM 调用 A 和调用 B **各自独立兜底**。
- 调用 B（安全旁路）不可用时 → 降级为纯本地敏感词过滤（FR-5.2），标记 `safety_degraded=true` 但主流程不中断。

## §4 数据完整性

> 详见产品需求文档 §10 系统边界约束、FR-6.2；业务流设计文档 §3.4 并发与一致性策略

### 4.1 拉链表设计

> 详见产品需求文档 FR-6.2

- 用户画像、标签、偏好、安全档案、摘要记忆**均使用拉链表**。
- 拉链表通过 `parent_id` 链接历史记录，实现数据变更可追溯、可回滚。

### 4.2 系统边界约束（全量 19 项）

> 详见产品需求文档 §10

以下列出所有 P0/P1 优先级的边界约束场景及处理策略：

**推荐结果边界**:

| 边界场景 | 优先级 | 处理策略 | 关联 FR |
|---------|:---:|---------|--------|
| 全过滤冲突（安全档案过滤全部候选） | 🔴 P0 | 按 `allergen_severity` 排序，选最低值候选 + 显式标注所有已知过敏原。**不**推送零过滤推荐 | FR-3.5 |

**数据一致性边界**:

| 边界场景 | 优先级 | 处理策略 | 关联 FR |
|---------|:---:|---------|--------|
| 长期不活跃回归（> 3 个月） | 🟡 P1 | 轻量重新引导（<= 2 题）+ 选"生活大变化"时降权 50%。回归后首次推荐不触发冷归档查询（冷归档仅在第 2 轮后按需启动） | — |
| "被遗忘权"技术实现 | 🟡 P1 | 删除链路更新: Layer 1（即时清除）→ Layer 2 热检索（软删除，30 天物理删除）→ Layer 2 冷归档（软删除标记，30 天后物理删除索引文件和原文）→ Layer 3（立即软删除）→ 全量会话原文（软删除） | FR-9.4 |

### 4.3 并发与一致性策略

> 详见业务流设计文档 §3.4

| # | 场景 | 策略 | 细节 |
|---|------|------|------|
| 1 | **多标签页同用户** | 独立 session_id，Layer 2/3 共享 | 每个标签页独立 Layer 1 窗口。Layer 3 读取共享、写入乐观锁 |
| 2 | **压缩任务写摘要** | 外部存储先写 → Redis 后写 | 压缩任务独立于 session 生命周期。session 已关闭仍可写入 |
| 3 | **用户同时精炼 + 发新消息** | FIFO 排队 + 合并检测 | 两条精炼指令不同维度/同方向 → 合并为 1 次。反方向 → 分别执行。合并上限: 2 条 |
| 4 | **画像异步更新 vs 实时读取** | 读取最新版本号，不阻塞 | M3 读取时用快照版本。M1 异步更新不影响当前生成。下次请求取新版本 |
| 5 | **冷归档查询与热检索并发** | 热检索先返回，冷归档异步追加 | M4 同时发起两条查询。热检索 < 20ms 即刻返回。冷归档下一轮注入 |
| 6 | **SSE 推送与前端状态** | generation_id + phase 协议 | 前端 localStorage 绑定 generation_id → phase，断开后从断点续 |
| 7 | **Layer 1 溢出压缩** | 原子操作，压缩期间 Layer 1 只读 | 压缩任务获取写锁。若期间新消息到达 → 先排队，压缩完成后写入 |

### 4.4 全量会话持久化

> 详见产品需求文档 FR-4.11

- 全量对话原文持久化 + 按月分区 + 索引优化。
- 情绪标签从 metadata JSON 冗余存储为独立列 `emotion_tags`，支撑列表查询性能。
- 历史会话列表默认**按天聚合**——同一天的多个微会话折叠为一条记录。
- 全量会话写入后不可篡改（append-only）。

### 5.2 API 通用规范

> 详见业务流设计文档 §4.3

| 规范项 | 约定 |
|--------|------|
| **认证** | Bearer Token（JWT）。游客使用临时 token（browser_id 签发，30 天有效） |
| **请求/响应格式** | `Content-Type: application/json`，SSE 为 `text/event-stream` |
| **错误格式** | `{"error": {"code": "ERROR_CODE", "message": "...", "retryable": true/false, "details": {...}}}` |
| **分页** | 游标分页 `?cursor={last_id}&limit=20`（适合动态数据） |
| **幂等键** | 写操作请求头 `Idempotency-Key: {uuid}`（防重复提交） |
| **URL 版本** | 路径前缀 `/api/v1/...` |
| **限流** | 写操作 30 req/min/用户；读操作 120 req/min/用户 |
| **trace_id 响应头** | 响应头 `X-Trace-Id` 返回，前端 console 打印，方便问题追溯 |

### 5.4 缓存主动失效策略

> 详见产品需求文档 FR-6.4

| 变更事件 | 失效缓存 | 机制 |
|---------|---------|------|
| 知识图谱新增节点/边 | GraphRAG 热点子图 | 图谱写入时发布 Redis Pub/Sub → 标记对应 key 过期 |
| 用户更新安全档案 | 外部偏好资源 | Session 内立即刷新（不等到 TTL） |
| 通感映射新增（人工审核后） | 通感解码缓存对应 key | 运营后台操作时主动删除 |
| 用户画像标签重大变更（>= 3 个标签同时变） | 用户画像预热缓存 | 异步任务完成后失效 |
| Layer 2 热检索新摘要写入 | 无（仅追加） | 摘要只追加不修改，无需失效 |

- 游客注册成为正式用户时，对话和配方状态自动迁移至新账号。
