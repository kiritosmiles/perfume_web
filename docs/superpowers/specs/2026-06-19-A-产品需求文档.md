## 1. 项目概述

> 详见技术需求文档 §1 项目概述

### 1.1 项目定位

**情绪人格×香水 AI Agent** 是一款面向 C 端用户的智能香氛推荐系统。系统以大语言模型（LLM）为核心推理引擎，结合 GraphRAG 图结构推理、情感分析和三层分层记忆架构，将用户的情绪状态、人格特征、使用场景与香调知识图谱进行匹配，生成个性化香氛配方与文字解读，实现"情绪驱动型香水推荐"。

系统具备**会话意图感知**能力——自动识别用户当前是"为自己挑选"还是"为他人送礼"，并据此切换不同的上下文数据加载策略和推荐逻辑。

当前以 **Web 部署**为主，架构预留 App 扩展能力。

### 1.2 项目背景

近几年"情绪价值"已从互联网热词演变为真实的消费驱动力。香水品类正在成为"情绪消费"中最典型、增长最快的品类之一。送礼是香水消费的核心场景——大量购买发生在节日、纪念日、生日等情境中。

### 1.3 C 端核心设计原则

| 原则 | 说明 | 设计体现 |
|------|------|---------|
| **低门槛** | 用户不需要懂香调术语 | 游客模式、情绪卡片、场景化引导 |
| **低延迟** | 首字优先 | 全链路并行化、预加载、骨架流式 |
| **低摩擦** | 减少操作负担，隐式优于显式 | 隐式反馈、选项化反问、异步尾量 |
| **被理解感** | 让用户感到"被听见" | 情绪确认微交互、精炼对话、主动回访 |
| **可掌控** | 用户知道 AI 记住了什么 | 记忆透明化、安全档案维护、意图手动切换 |
| **安全感** | 不滥用情绪数据；危机正确响应 | 本地敏感词、危机差异化、数据删除权 |
| **场景智能** | 感知"为谁挑选" | 自用/送礼/探索三种意图差异化策略 |
| **可回溯** | 用户可回顾自己的情绪旅程 | 全量会话持久化、历史浏览、情绪时间线 |

### 1.4 项目目标

| 编号 | 目标 | 说明 |
|------|------|------|
| G1 | 人格型香水 Agent | 基于用户稳定人格画像推荐香调 |
| G2 | 情绪陪伴型香水 Agent | 基于用户当前情绪状态推荐香调，提供情绪陪伴 |
| G3 | 五维用户建模 | 记忆、情绪、身份、社交、人格五维度 |
| G4 | AI 配方生成 | LLM + GraphRAG 生成香氛配方，经调香师调制后输出香氛卡片 |
| G5 | AI 赋能情绪+ | 以情绪价值为方向，持续探索 AI 赋能场景 |
| G6 | 意图感知推荐 | 自动识别自用/送礼/探索意图 |
| G7 | 分层记忆系统 | 三层记忆架构：会话记忆 → 摘要记忆 → 外部偏好 |

### 1.5 交付物

- **香氛卡片**: 系统生成配方 → 调香师（Helen Yee）调制 → 实体香氛卡片
- **智能推荐**: 对话式交互，产品推荐、对比、介绍；自用和送礼两套链路
- **会话历史**: 用户可浏览全部历史对话原文及对应推荐结果

## 2. 用户群体

### 2.1 用户状态机

> 详见业务流设计文档 §2.2 用户状态机

```
                      ┌──────────────┐
                      │    游客       │
                      │   (Guest)    │
                      └──┬───┬───┬──┘
                         │   │   │
             注册         │   │   │ 30天未注册 → 数据物理删除
                         │   │   │
              ┌──────────┘   │   └──────────┐
              ▼              ▼              ▼
       ┌──────────┐   ┌──────────┐   ┌──────────┐
       │  新用户   │   │  游客离开  │   │  游客回归  │
       │ (New)    │   │  无转化    │   │  再对话    │
       └────┬─────┘   └──────────┘   └──────────┘
            │
            │ 第 4 次对话起
            ▼
     ┌──────────┐        > 7 天未访问        ┌──────────┐
     │  老用户   │ ────────────────────────→ │  不活跃   │
     │ (Regular) │ ←──────────────────────── │ (Dormant) │
     └────┬─────┘    回来继续对话             └────┬─────┘
          │                                       │
          │                              > 3 个月未访问
          │                                       │
          │         ┌──────────┐                  │
          │         │  回归用户  │ ←───────────────┘
          │         │ (Returning)│   轻量重新引导 ≤2题
          │         └────┬─────┘   选"生活大变化"→降权50%
          │              │         冷归档第2轮后才启动
          │   连续 3 次     │
          │   overstep      │
          ▼                ▼
    ┌──────────┐    ┌──────────┐
    │  建议转人工 │    │  回到     │
    │  或新Session│   │  Regular  │
    └──────────┘    └──────────┘
```

| 状态 | 定义 | 进入条件 | 退出条件 |
|------|------|---------|---------|
| `guest` | 游客 | 首次访问无 cookie | 注册 / 30 天过期 |
| `new` | 新用户 | 注册成功 | 第 4 次对话 |
| `regular` | 老用户 | 从 new 升级 / 从 dormant 恢复 | 连续 7 天未访问 |
| `dormant` | 不活跃 | 连续 7 天未访问 | 用户回来 / 回访通知触发 |
| `returning` | 回归 | 连续 3 个月未访问后回来 | 轻量引导完成后 → regular |

### 2.2 游客模式

> 详见技术需求文档 FR-1.7 游客模式

- 免注册完整体验一次自用推荐
- 不创建持久化画像
- 不启用送礼模式
- 游客对话存储在临时表（`temp_conversations`），通过设备指纹 / browser_id 关联
- 游客注册成为正式用户时，对话和配方状态自动迁移至新账号
- 游客 30 天未注册 → `temp_conversations` 中对应记录自动物理删除

### 2.3 渐进式画像构建

> 详见技术需求文档 FR-1.3 渐进式画像构建

- 前 3 次对话轻量模式（情绪即时响应为主）
- 第 4 次起切换完整推理链路

### 2.4 用户分级与用量配额

> 详见业务流设计文档 §5 用户分级与用量配额

| 维度 | 游客 (Guest) | 免费用户 (Free) | 付费用户 (Premium) |
|------|:---:|:---:|:---:|
| **对话次数** | 1 次完整会话 | 10 次会话/天 | 不限 |
| **推荐生成** | 1 次 | 15 次/天（快速+深度合计） | 不限 |
| **深度模式** | 不触发 | 3 次/天 | 不限 |
| **可用意图** | 仅 self_use | self_use / gift / explore | 全部 |
| **精炼** | 最多 3 轮 | 不限次（本地规则） | 不限 |
| **历史浏览** | 不持久化 | 最近 30 天 | 全量 |
| **卡片制作（调香师协作）** | 不可用 | 1 次/月 | 3 次/月 |
| **优先级** | — | 标准 | 高峰期 LLM 调用优先 |

配额耗尽时的 UX 设计：

| 场景 | 用户看到 | 用户可操作 |
|------|---------|-----------|
| 游客用完 | 「你的免费体验已结束」 | 注册 / 回首页 |
| 免费用户日会话耗尽 | 「今天的对话次数已用完，X 小时后重置」+ 倒计时 | 浏览历史 / 升级 |
| 免费用户日生成耗尽 | 「今天的推荐次数已用完」 | 浏览历史 / 精炼已有推荐 / 升级 |
| 免费用户深度耗尽 | 「深度探索本日已用完，为你使用快速推荐」 | 接受快速推荐 |
| 免费用户月卡片耗尽 | 「本月卡片制作次数已用完」 | 升级 / 下月再来 |
| 接近限额（80%） | 淡色提示「今日还可生成 X 次」 | 无操作，仅提示 |

## 3. 核心功能

### 3.1 用户画像与人格建模

> 详见技术需求文档 §2 业务模块一

#### 3.1.1 用户画像建档 (FR-1.1)

系统为新用户自动创建画像档案。输入：注册信息、初次对话、情绪识别结果、引导问卷、安全档案。输出：结构化用户画像（人格标签、情绪倾向、偏好香调、社交风格等）。

#### 3.1.2 冷启动引导问卷 (FR-1.2)

极简场景化问卷（≤ 3 题），用场景替代术语。每个选项背后预设映射向量，无需 LLM，毫秒级响应。

#### 3.1.3 渐进式画像构建 (FR-1.3)

前 3 次对话轻量模式（情绪即时响应为主）；第 4 次起切换完整推理链路。

#### 3.1.4 社交授权导入 (FR-1.4)

一键授权导入第三方兴趣/情绪标签（网易云音乐、小红书等）。

#### 3.1.5 人格建模 — 三种 Session 模式 (FR-1.5)

Identity / Context / Novelty。延迟判断 + 用户显式切换（「今天的我」「一直的我」「想尝试的我」）。

#### 3.1.6 动态标签更新 (FR-1.6)

每次对话结束后异步处理、用户反馈后、周期性批量任务。

#### 3.1.7 游客模式 (FR-1.7)

免注册完整体验一次自用推荐；不创建持久化画像；不启用送礼模式。详见 [§2.2 游客模式](#22-游客模式)。

#### 3.1.8 香氛安全档案 (FR-1.8)

仅在会话意图 = `self_use` 时生效。独立外部偏好资源存储，不嵌入用户画像主表。新增 `allergen_severity` 字段（1-5 整数），支撑"全过滤冲突"场景下按致敏严重度排序选择最低风险推荐。

#### 3.1.9 游客数据迁移 (FR-1.9)

- 游客注册成为正式用户时，其在游客模式下的对话和配方状态自动迁移至新账号
- 迁移机制：根据 `browser_id` 将 `temp_conversations` 中的对话记录迁移至用户正式 `conversations` 表
- 清理：游客 30 天未注册 → `temp_conversations` 中对应记录自动物理删除

### 3.2 情绪识别与输入

> 详见技术需求文档 §3 业务模块二

#### 3.2.1 双通道情绪输入 (FR-2.1)

- **通道 A**: 自由文本输入
- **通道 B**: 情绪卡片（多选，最多 2 张，预设向量）

#### 3.2.2 情绪识别 + 意图 + 通感 (FR-2.2)

采用**双路径分级**策略——高频路径走 BERT 轻量分类（~50ms），低频路径走 LLM 兜底（~800ms）。

```
用户文本输入
    │
    ├──→ 路径A: BERT 情绪分类 (主路径,覆盖 ~70% 对话)
    │     延迟: ~50ms (GPU推理) / ~100ms (CPU推理)
    │     输出: 情绪分布向量 {"焦虑": 0.6, "平静": 0.2, ...}
    │
    ├──→ 路径B: LLM 通感+意图兜底 (仅 ~30% 对话)
    │     触发: BERT 置信度 < 0.7 或 检测到通感表达 或 送礼关键词
    │     延迟: ~800ms (热启动)
    │     输出: 完整结构化输出 (情绪修正 + 意图 + 通感)
    │
    └──→ 关键词意图检测 (始终执行,< 1ms)
          送礼关键词命中 → 直接标记 gift
```

#### 3.2.3 情绪确认微交互 (FR-2.3)

确认文案展示检测到的情绪和意图。送礼意图时切换为收礼人引导语气。用户可在 3s 内点「不对」修正。

#### 3.2.4 情绪向量化 (FR-2.4)

将检测到的情绪输出为 8 类情绪分布向量，和为 1。

#### 3.2.5 情绪价值维度映射 (FR-2.5)

六大维度映射：愉悦度、激活度、支配度、社交性、审美性、怀旧感。

#### 3.2.6 混合情绪支持 (FR-2.6)

多标签情绪分布向量表达混合情绪状态。

#### 3.2.7 场合/场景输入 (FR-2.7)

6 个预设场景：通勤工作、约会之夜、宅家放松、聚会社交、挑选礼物、随便看看。

#### 3.2.8 环境感知 (FR-2.8)

天气/季节/时段感知，融合权重 0.1~0.2（极端天气时升至 0.2），不受意图影响。

#### 多源融合推理流程

```
情绪输入(A/B) + 场景选择 + 环境信号
         ↓
   多源向量融合 (情绪 0.6 / 场景 0.2~0.3 / 环境 0.1~0.2)
         ↓
   通感检测 → 如有: 解码香调作为 GraphRAG 检索引导信号
         ↓
   意图门控 → 决定上下文加载策略
         ↓
   三层记忆检索
         ↓
   GraphRAG 推理 → 配方生成
```

### 3.3 香调推荐与配方

> 详见技术需求文档 §4 业务模块三

#### 3.3.1 情绪—香调映射关系 (FR-3.1)

建立情绪向量到香调空间的映射。送礼场景增加"礼物接受度"权重。

#### 3.3.2 AI 配方生成（意图门控）(FR-3.5)

```
if session_intent == "self_use":
    加载 Layer 3 安全档案 → 过滤禁忌 → 生成
    全过滤冲突时: 按 allergen_severity 排序,选最低值候选 + 显式标注
elif session_intent == "gift":
    跳过安全档案 → 生成 + 常见过敏原标注
elif session_intent == "explore":
    跳过安全档案 → 最大化多样性
```

#### 3.3.3 产品输出与对比 (FR-3.6)

产品推荐、多维度对比、故事性介绍文案。

#### 3.3.4 推荐精炼对话 (FR-3.7)

用户对推荐结果提出调整需求时，系统以最低延迟完成局部调整。精炼不走完整推理链路，只在当前候选集上做局部操作。

**精炼流程**:

```
Turn N: 用户精炼请求
    │
    ├── 第 1 次精炼: 规则引擎直接处理 (< 500ms)
    │   规则覆盖维度:
    │   - 甜度 ("太甜"/"不够甜")
    │   - 浓度 ("太浓"/"太淡")
    │   - 清新度 ("更清新"/"太沉闷")
    │   - 性别倾向 ("更女性化"/"更男性化")
    │   - 温暖度 ("太冷"/"更温暖")
    │   - 香调切换 ("换个香调"/"试试花香")
    │   - 场合切换 ("更正式"/"更休闲")
    │   - 年龄适配 ("太年轻"/"太成熟")
    │   - 季节适配 ("更适合夏天用")
    │   不可处理: "太贵了"(价格) / "前调太冲"(三调精细调整)
    │
    ├── 第 2 次精炼:
    │   ├── 先尝试规则引擎解析
    │   └── Agent Gate: 语义判断 (轻量 LLM, 200ms):
    │       ├── 规则可处理 → 继续规则精炼 (< 500ms)
    │       ├── 语义不满("没理解我") → 停止精炼，降级到追问
    │       └── 超时 → 继续规则精炼（保守行为）
    │
    └── 第 3 次精炼:
        ├── 规则引擎再次尝试
        └── 依然解析失败 → 自动升级深度模式
            文案：「你已经调整了 3 次——让我换个思路，从不同的角度重新为你搭配」
            深度模式产生的推荐作为全新推荐起点，精炼计数器重置
```

精炼退出机制：深度模式后如仍不满意 → 收藏+稍后继续模式，展示所有已尝试版本。

精炼前的意图再确认：如果上一轮和当前轮的 LLM 意图判定不一致，立即触发轻量确认——不等待第 2 次精炼的 Gate。

#### 3.3.5 推荐多样化 (FR-3.8)

显式「换个风格」按钮 + 自动多样性注入 + 滑块控制。跨风格探索（目标风格与当前推荐在香调空间中的余弦距离 ≥ 0.6）触发深度模式进行多角度探索。

#### 3.3.6 送礼场景推荐策略 (FR-3.9)

- 收礼人引导（≤ 2 题）+ 经典礼物兜底 + 礼物叙事文案 + 过敏原标注
- 收礼人零信息: 通用礼物香调 Top 5 卡片墙
- 收礼人是儿童（< 12 岁）: 温和配方 + 去性感化文案 + 无酒精提示

#### 3.3.7 调香师协作桥 (FR-3.10)

在配方生成的数字世界与实物调制的物理世界之间建立最简连接。MVP 范围：
- 用户确认配方并选择"制作香氛卡片" → 配方数据写入调香师工作队列
- 调香师（Helen Yee 团队）查看队列，标记状态：`accepted` / `needs_adjustment` / `rejected`
- 状态变更时，系统向用户推送通知（Web Notification / 邮件）
- 用户可在"我的卡片"中查看进度：`pending` → `accepted` → `in_progress` → `completed`

### 3.5 对话智能与安全

> 详见技术需求文档 §6 业务模块五

#### 3.5.1 转人工机制 (FR-5.3)

当系统无法安全/有效地继续自动对话时，提供明确的人工介入路径。

| # | 触发条件 | 严重度 | 行为 |
|---|---------|:---:|------|
| 1 | 连续 3 次 Agent 角色越界 (`overstep`) | 高 | 强制转人工：系统停止自动回复，展示人工联系入口 |
| 2 | 危机检测 severity = `high` | 高 | 展示热线 + 人工联系入口 |
| 3 | 用户显式请求（"转人工""找客服"） | 中 | 立即转接，不等其他条件 |
| 4 | 连续 5 轮精炼仍不满意 | 中 | 建议转人工 |
| 5 | 同一 session 内 ≥ 3 个 LLM 调用降级 | 中 | 主动提示联系调香师团队 |

MVP 转接目标: 调香师团队邮箱/留言表单。转接时自动生成"转接摘要"（当前 session 摘要 + 最近推荐结果 + 触发原因），用户无需重复描述问题。

#### 3.5.2 过渡动画与等待体验 (FR-5.5)

长时间等待态（> 3s）: 香料知识趣味卡片轮播，降低用户焦虑。

#### 3.5.3 危机场景差异化交互 (FR-5.6)

当检测到用户处于心理危机状态时，系统立即切换交互模式——暂停推荐、避免轻浮语气、引导支持资源。

危机检测机制（三层）：
- **第一层**: 危机关键词字典（本地, < 1ms，P0 底线保障）
- **第二层**: BERT/LLM 语义兜底（异步，增加 crisis 判定维度）
- **第三层**: 误判管理（用户否认时记录为误判样本）

危机响应流程：
- 立即暂停当前推荐生成（SSE 中断）
- 开场白切换为关怀语气（禁止推荐任何香调）
- 展示可配置的心理援助热线（按地区匹配）
- 本轮 session 禁止启动任何推荐
- 用户仍可自由对话，但不主动提供推荐
- 用户手动切换意图标签或新 session 后恢复正常

#### 3.5.4 会话意图检测 + 信息完整性判定 (FR-5.7)

三层判定流程：
1. **关键词检测** (< 1ms): "给自己买" → self_use / "送给" "礼物" → gift
2. **LLM 调用 A 合并判定** (~800ms): 含意图+情绪+通感+信息完整性维度
3. **用户显式切换**: 前端意图标签（为自己/为他人/随便看看），用户可随时点击切换

信息完整性维度输出:
- `sufficient`: 核心维度齐备 + GraphRAG 候选 ≥ 5
- `partial`: 1-2 个维度缺失，但工具可补充
- `insufficient`: 核心维度缺失且工具无法补充

#### 3.5.5 抽象/超现实需求处理 (FR-5.8)

LLM 通感解码 → 香调转译 → 确认微交互。通感解码后的香调词汇作为 GraphRAG 检索的引导种子节点。

示例: "下雨的味道" → 解码为 [臭氧调, 湿润木质, 青草] → 以这三个节点为种子在 Neo4j 中扩散 2 跳 → 收集关联香调和原料 → 融入推理。

#### 3.5.6 Agent 角色边界保护 (FR-5.9)

越界检测（`normal` / `borderline` / `overstep`）+ 分层话术响应。连续 3 次 `overstep` → 强制建议联系人工。

#### 3.5.7 信息完整性 Agent Gate (FR-5.11)

在推理生成前插入一个有硬边界的决策节点——判断当前上下文是否足够支撑高质量推荐，不够则限时收集信息或整理追问。

**判定规则**:

| 条件 | 判定 | 行为 | 示例 |
|------|:---:|------|------|
| 所有核心维度齐备 + GraphRAG 候选 ≥ 5 | `sufficient` | 跳过 Gate，0ms 开销 | 老用户自用推荐 |
| 1-2 个维度缺失，但工具可补充 | `partial` | 调 1 次工具(≤200ms)，补上 → 生成 | 送礼场景，送礼历史存在 |
| 核心维度缺失且工具无法补充 | `insufficient` | 整理追问清单（≤ 3 个问题），直接输出给用户 | 送礼场景，收礼人完全未知 |
| LLM 超时(500ms) | — | 当作 `sufficient`，用已有信息生成 | 网络抖动 |

**与意图的联动**:

| 意图 | sufficient 条件 | insufficient 触发 | 追问方向 |
|------|:---:|------|------|
| `self_use` | 情绪已知 + 场景已知 + 安全档案已加载 | 场景标签完全未知 | "日常用还是特定场合？" |
| `gift` | 收礼人身份标签 ≥ 2 项 + 场景 | 收礼人完全未知(0 标签) | "收礼人年龄/生活类型/用过的香水？" |
| `explore` | 探索方向已知 | 几乎不触发 | 探索模式允许信息稀疏 |


## 5. 分阶段交付计划

### 6.2 政策与合规背景

- 国务院《加快培育服务消费新增长点工作方案》（2026.02.02）：支持情绪式、体验式服务消费新场景。

**产品层面合规要点**:
- 用户数据加密（TLS + AES-256），加密密钥使用云厂商 KMS 管理
- 情绪数据保护（敏感信息 + 危机场景独立权限）
- 用户授权机制（明确告知数据采集范围，含全量会话存储、冷归档永久保留，获取显式同意）
- 数据删除权（全量数据删除含冷归档层；30 天软删除 + 物理删除）
- 审计日志（全操作 + 偏好变更 + 意图切换 + 摘要写入/淘汰 + 敏感词库变更 + KMS 访问）
- 第三方依赖最小化（本地敏感词 + 城市级天气）

## 7. 局限性与未来方向

> 详见技术需求文档 §17 项目局限性与未来方向

### 7.1 已知局限性

| 局限性 | 缓解措施 |
| ------ | -------- |
| 香味感知主观性 | 精炼对话 + 记忆透明化 |
| 数据隐私风险 | 本地审核 + 用户授权 + 数据删除权 |
| 冷启动问题 | 引导问卷 + 社交导入 + 渐进画像 + 游客模式 |
| 情感识别准确率 | 情绪确认微交互 + 用户修正 |
| 敏感词本地化覆盖面 | 周期性维护 + 社区举报 |
| 危机识别误判 | 偏向"宁可误判、不可漏判" |
| 送礼意图误判 | 置信度门控 + 反问确认 + 精炼前再确认 |
| 送礼场景冷启动 | 经典礼物香调兜底 |
| 抽象需求解码准确率 | LLM 通感转译 + 用户确认微交互 |
| Layer 2 双写延迟 | 外部先写 + Redis 后写 + 冷归档兜底 |
| 记忆衰减模型粗糙 | 固定值，未来升级为个性化衰减率 |
| 冷归档查询延迟（< 1s） | 异步追加上下文，不阻塞主流程 |
| 调香师协作桥为手动流程 | 后续版本可实现调香师端完整 UI |

### 7.2 未来研究方向

- **多模态情绪识别**: 语音语调、面部表情等
- **推荐可解释性**: 可视化推理路径
- **用户反馈训练闭环**: 利用隐式+显式反馈持续优化推荐
- **B 端调香师协作平台**: 完整的生产管理、物流跟踪
- **收礼人画像跨 session 持久化**: 避免每次送礼都要重新描述收礼人
- **意图检测模型在线学习**: 基于用户修正反馈持续改进
- **个性化记忆衰减率**: 从固定值升级为学习型衰减率
- **App 扩展**: 原生推送、离线、原生分享
- **全球化**: IFRA 合规、文化映射、多语言通感解码
- **冷归档实时索引升级**: 从每日重建 → 近实时更新
- **交易链路集成**: 购物车、支付、订单管理
