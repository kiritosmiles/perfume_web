## §2 LLM 调用架构

### 2.1 LLM 调用策略总览

> 详见产品需求文档 §1.8

系统采用**分层 + 双路径 + 约束化 Agent Gate**策略 -- 输入层 BERT 快路径 / LLM 兜底、推理层按需深浅、精炼层规则+语义 Gate、每层有硬边界。

```
                       用户消息到达
                            │
    ┌───────────────────────┴───────────────────────┐
    │  输入处理层 (始终执行)                           │
    │                                                 │
    │  ┌─ 情绪卡片通道(67%): 预设向量,< 5ms ────────┐ │
    │  │  意图: 关键词 < 1ms                        │ │
    │  └────────────────────────────────────────────┘ │
    │                                                 │
    │  ┌─ 文本通道(33%): BERT 主路径(70%) + LLM兜底   │ │
    │  │  ├─ BERT 情绪分类: ~50ms                    │ │
    │  │  │   置信度>=0.7 -> 直接使用,跳过LLM           │ │
    │  │  └─ LLM 兜底(30%): 通感解码+意图,~800ms     │ │
    │  └────────────────────────────────────────────┘ │
    │                                                 │
    │  └──-> 调用 B: 安全链路 (异步旁路, LLM)           │
    └───────────────────────┬───────────────────────┘
                            │
                      意图门控 + 记忆检索
                            │
                            ▼
    ┌─────────────────────────────────────────────┐
    │  Agent Gate: 信息完整性检查 (v2.6)        │
    │  预算: 500ms + 1 次重试                      │
    │  ├─ sufficient -> 跳过 (0ms, ~95% 场景)    │
    │  ├─ partial -> 调工具补(200ms) -> 生成      │
    │  └─ insufficient -> 整理追问 -> 输出用户     │
    └───────────────────────┬─────────────────────┘
                            │
    ┌───────────────────────┴───────────────────────┐
    │  推理生成层 (快速模式 + 深度模式)                │
    │                                                 │
    │  ┌─ 快速模式 (默认, ~88% 场景) ─────────────┐  │
    │  │  单次 LLM + Prompt Cache                   │  │
    │  │  骨架: 情绪卡片 ~1.3s / 文本 ~1.5s         │  │
    │  └──────────────────────────────────────────┘  │
    │                                                 │
    │  ┌─ 深度模式 (~12% 高价值场景) ─────────────┐  │
    │  │  Supervisor(规则引擎) -> 2xSubagent并行    │  │
    │  │  Graph(纯检索) + Composer(LLM骨架先行)     │  │
    │  │  骨架: ~1.7s    完整: ~2.7s               │  │
    │  └──────────────────────────────────────────┘  │
    └───────────────────────┬───────────────────────┘
                            │
                        推荐输出
                            │
    ┌───────────────────────┴───────────────────────┐
    │  精炼层                                           │
    │  ├─ 第1次精炼: 规则引擎 (< 500ms)                 │
    │  ├─ 第2次精炼: Agent Gate 语义判断 (200ms)     │
    │  │   ├─ 规则可处理 -> 继续精炼                      │
    │  │   └─ 语义不满("没理解我") -> 降级追问            │
    │  └─ 第3次精炼: 规则再尝试 -> 仍失败 -> 升级深度模式  │
    └─────────────────────────────────────────────────┘
```

- v2.6 核心新增: Agent Gate: 信息完整性检查（推理前，500ms 硬边界，95% 场景跳过）; Agent Gate: 精炼语义理解（第 2 次精炼后，200ms 硬边界）
- v2.5.1 核心优化: BERT 替代 LLM 做情绪分类（70% 文本对话，从 LLM 1.5s -> BERT 50ms）; Supervisor 去 LLM 化（策略矩阵硬编码，< 10ms）; Graph Explorer 纯检索模式（非 Agent，无工具调用往返）; 加权平均骨架延迟 ~1.37s
- 调用 A 和 B 均使用结构化输出（JSON Schema）
- 有效 LLM 调用次数从 v2.5 的 7 次降至 4 次（BERT 情绪 + 1 LLM 快速配方 + 2 安全旁路）。深度模式额外 +3 次（Composer + Safety + Aggregator），仅 ~12% 场景
- Agent Gate 在最坏情况额外 +1 次轻量 LLM 调用（信息不足时追问决策），仅 ~3% 场景；精炼语义 Gate 仅第 2 次精炼触发

### 2.2 LLM 调用约束矩阵

> 详见产品需求文档 §1.9

**设计原则**: 每次 LLM 调用必须有硬边界 -- 超时即降级，不等待；最多 1 次重试；耗尽后不兜圈。

| # | 调用点 | 类型 | 正常延迟 | 超时 | 重试 | 重试策略 | 耗尽后行为 | 覆盖场景 |
|---|--------|------|:---:|:---:|:---:|---------|-----------|------|
| 1 | BERT 情绪分类 | 本地推理 | 50ms | 200ms | 0 | -- | 走 2 LLM 兜底 | 100% 文本对话 |
| 2 | LLM 情绪兜底(含通感+意图) | 云端 LLM | 800ms | 1.5s | 1 | 换轻量模型(500ms) | 用 BERT 结果(即使低置信) | ~10% 文本对话 |
| 3 | 调用 B 安全旁路 | 云端 LLM(异步) | 不阻塞 | 2s | 1 | 换轻量模型 | 降级纯本地敏感词,标记 `safety_degraded` | 100% 对话 |
| 4 | 快速模式 LLM 生成 | 云端 LLM | 700ms | 2s | 1 | 换轻量模型(500ms) | 配方骨架缓存兜底(预计算) | ~88% 场景 |
| 5 | Deep: Composer | 云端 LLM | 700ms | 1.5s | 1 | 换轻量模型 | Graph 纯检索结果作为最终推荐 | ~12% 场景 |
| 6 | Deep: Safety Reviewer | 云端 LLM(轻量) | 400ms | 800ms | 0 | -- | 跳过审查，标记 `unreviewed` | ~12% 场景 |
| 7 | Deep: Aggregator | 云端 LLM(轻量) | 600ms | 1s | 1 | 换轻量模型 | 规则排序(图谱分数+安全分数+多样性加权) | ~12% 场景 |
| 8 | Agent Gate: 信息完整性 | 云端 LLM(轻量) | 200ms | 500ms | 1 | 换轻量模型(200ms) | 当作 `sufficient`，用已有信息生成 | ~5% 场景触发 |
| 9 | Agent Gate: 精炼语义 | 云端 LLM(轻量) | 200ms | 500ms | 0 | -- | 继续规则精炼（不追问） | ~15% 精炼对话 |

**重试与超时细则**:

| 约束维度 | 规范 |
|---------|------|
| **超时判定** | 从 LLM 请求发出开始计时。到达 timeout 毫秒后无响应 -> 视为超时，不等结果返回 |
| **重试间隔** | 0ms（立即重试）。理由：超时已说明当前模型不可达，等待无益 |
| **重试模型切换** | 主模型超时 -> 切换到备用轻量模型（如从 Sonnet 4.6 -> Haiku 4.5 或 Qwen-7B 微调版）。轻量模型 token 限制减半 |
| **0 次重试的调用** | 1/6/9: 1 是本地推理不涉及云端；6 是非主链路的尾量审查，跳过不影响安全基线；9 语义不满判断失败时保守行为是"继续精炼"而非追问 |
| **重试也失败** | 执行"耗尽后行为"列中的 degrade 策略。降级输出必须带标记（如 `quality_degraded=true`）供监控 |
| **全链路累计** | 单次用户请求的 LLM 调用总时长（含重试）上限为 **10s**。超过 -> SSE 中断并输出"抱歉，我正在经历高负载，请稍后重试"。深度模式最坏情况: 2(2.0s)+5(2.0s)+7(1.5s)+8(0.7s) ~ 6.2s，加上渲染和网络 RTT 余量。异步调用 3 不计入累计 |

### 2.4 BERT 情绪分类 -- 双路径架构

> 详见产品需求文档 FR-2.2

```
用户文本输入
    │
    ├──-> 路径A: BERT 情绪分类 (主路径,覆盖 ~70% 对话)
    │     模型: bge-emotion / 自训练多标签分类器
    │     延迟: ~50ms (GPU推理) / ~100ms (CPU推理)
    │     输出: 情绪分布向量 {"焦虑": 0.6, "平静": 0.2, ...}
    │
    ├──-> 路径B: LLM 通感+意图兜底 (仅 ~30% 对话)
    │     触发: BERT 置信度 < 0.7 或 检测到通感表达 或 送礼关键词
    │     延迟: ~800ms (热启动)
    │     输出: 完整结构化输出 (情绪修正 + 意图 + 通感)
    │
    └──-> 关键词意图检测 (始终执行,< 1ms)
          送礼关键词命中 -> 直接标记 gift
```

- **路径选择逻辑**: BERT 置信度 >= 0.7 且无通感关键词 -> 纯 BERT 结果，跳过 LLM。BERT 置信度 < 0.7 或检测到通感模式 -> 启动 LLM 路径 B，以 BERT 结果作为 few-shot 锚点
- **通感触发精化**: 通感关键词（"像""味道""感觉"）必须与抽象名词共现才触发。匹配模式: `{像|好像|仿佛} + {一种|某种} + {抽象名词}` 或 `{抽象名词} + {的} + {味道|感觉|气息}`。单独出现"我感觉这个不错"不触发通感路径
- **BERT 模型选型**: `bge-large-emotion` 或基于中文情绪标注数据微调的 `bert-base-chinese`。输出 8 类情绪标签 + 强度
- **微调数据策略**: MVP Phase 1 优先使用零样本 `bge-large-emotion` 作为基线。>= 2 万条领域标注数据后再做微调。标注质量控制: LLM 标注结果由 2 名标注员交叉校验（Cohen's Kappa >= 0.7），分歧样本由第三人仲裁
- **BERT 部署方案**: MVP Phase 1 使用 CPU 推理 (ONNX Runtime, 4 核, ~100ms)，延迟仍在可接受范围；GPU 实例根据文本对话占比增长后续评估

### 2.5 深度模式 -- Supervisor+Subagent 架构

> 详见产品需求文档 FR-3.12

- **触发条件**（任一满足即触发）:

| # | 触发条件 | 场景 | 说明 |
|---|---------|------|------|
| 1 | **精炼 3 轮后仍不满意** | 用户连续 3 次调整后仍有修改意愿 | FR-3.7 自动升级 |
| 2 | **跨风格大跳跃** | 用户点击"换个风格"且目标风格余弦距离 >= 0.6 | FR-3.8 触发 |
| 3 | **复杂送礼** | 送礼模式 + 用户不了解收礼人 + 收礼人特征自相矛盾 | -- |
| 4 | **情绪-场景冲突** | 用户情绪和场景需求存在天然张力 | 如"焦虑但需要社交场合" |
| 5 | **用户显式请求** | 用户表达"给我更多选择"、"从不同角度想想" | 主动触发 |

```
Supervisor (规则引擎,非LLM) // 去 LLM 化,< 10ms
  输入: 触发条件 + 用户上下文
  输出: 探索策略(硬编码策略矩阵,见下方)
    │
┌───────并行 Phase 1 ───────┐
│                             │
│ Subagent 1: Graph Explorer  │  // 纯检索模式(非Agent),~500ms
│  Neo4j图多路径遍历          │    无 LLM 工具调用往返
│  输出: 3~5条候选+图谱证据   │
│                             │
│ Subagent 2: LLM Composer    │  // 骨架先行,~700ms
│  LLM 从零创作配方           │    仅输出前60 tok(香调组合)
│  输出: 2~3个创意配方骨架    │    热启动+Prompt Cache
│                             │
└─────────┬───────────────────┘
          │ max(500, 700) = 700ms
          ▼
骨架可见点 (~1.7s)
   用户看到: 多条不同角度的香调组合
          │
          ▼
┌───────串行 Phase 2 ───────┐
│ Safety Reviewer (LLM轻量)  │  // ~400ms
│  后置审查+冲突检测          │    可与前端渲染并行
└───────────┬────────────────┘
            ▼
┌───────串行 Phase 3 ───────┐
│ Aggregator (轻量LLM)       │  // ~600ms
│  冲突消解->交叉验证->排序    │    选择Top-1~3
│  推荐理由+备选方案         │
└───────────┬────────────────┘
            ▼
    完整输出 (~2.7s)
```

**Supervisor 策略矩阵（规则引擎, 非 LLM）**:

| 触发条件 | Supervisor 分派策略 | Graph Explorer 模式 | LLM Composer 模式 |
|---------|-------------------|--------------------|--------------------|
| 精炼3轮升级 | 保留当前配方作为参考锚点，Graph 沿"相邻"方向探测，Composer 做"对立"方向创作 | k=3 邻近扩散 | 锚点对立面创作 |
| 跨风格跳跃 | Graph 在远端簇中密集搜索，Composer 不受图谱约束 | k=5 远端 + 权重 0.8 | 全自由创作 |
| 复杂送礼 | Graph 强制遍历送礼安全路径，Composer 优先礼物友好型 | gift_safe 边优先 | 礼物文案先行 |
| 情绪-场景冲突 | Graph 于情绪/场景两条路径分别遍历并取交集 | 双向取交集 | 冲突调和 |
| 用户显式请求 | 最大化多样性: Graph k=5 + Composer k=3 | k=5 全探索 | k=3 全自由 |

**延迟预算**:

| Phase | 操作 | 延迟 | 说明 |
|-------|------|:---:|------|
| 输入层 | 调用A(BERT情绪 + LLM通感兜底) | ~800ms | BERT 主路径 |
| 记忆检索 | Layer 1+2+3+GraphRAG 并行 | ~100ms | -- |
| Supervisor | 规则引擎(策略矩阵查找) | < 10ms | 去 LLM 化 |
| Subagent 并行 | max(Graph纯检索 500ms, Composer骨架 700ms) | ~700ms | Graph非Agent模式 |
| **骨架可见** | = 800+100+10+700+100(渲染) | **~1.7s** | 云端现实可达值 |
| Safety Review | LLM 轻量审查 | ~400ms | 可与渲染并行 |
| Aggregator | LLM 汇聚排序 | ~600ms | -- |
| **完整输出** | = 1700+400+600 | **~2.7s** | 云端现实可达值 |

- **成本控制**: 单次深度模式 = 1(BERT) + 2(Subagent并行) + 1(Safety轻量) + 1(Aggregator轻量) = **3 次 LLM 调用**（Supervisor 去 LLM 化节省 1 次）
- **监控**: 深度模式触发率（目标 ~12%）、深度模式后满意度对比、深度模式延迟 P99。Agent Gate 触发率（目标 ~5%）、追问回复率

### 3.1 数据所有权矩阵

> 详见业务流设计文档 §3.3

### 3.2 读写分离与分库分表

> 详见产品需求文档 FR-6.1

系统采用读写分离架构：写操作路由至主库，读操作路由至只读副本。会话原文表按月分区，支撑高写入吞吐。

### 3.3 数据拉链表设计

> 详见产品需求文档 FR-6.2

用户画像、标签、偏好、安全档案、摘要记忆均使用拉链表。通过 `parent_id` 链接历史记录，实现数据变更追溯。

### 3.4 RAG 增量更新

> 详见产品需求文档 FR-6.3

知识图谱更新采用增量模式：人工审核后入库，每次更新写入拉链表，支持回滚。

### 3.5 Redis 缓存策略

> 详见产品需求文档 FR-6.4

**v2.4 缓存层级全景**:

| 缓存层 | 内容 | TTL 策略 | 说明 |
|--------|------|---------|------|
| Layer 1 会话记忆 | 1+5 滑动窗口对话 | Session 生命周期 | 无 TTL，session 结束后释放 |
| Layer 2 热检索 | 近期摘要 | baseTTL + randomTTL | 防雪崩 |
| Layer 2 冷归档 | 全量历史摘要 | 对象存储 + FAISS 索引（每日重建） | 永久保留 |
| 外部偏好资源 | 安全档案/不喜欢/送礼历史 | Session + 5min | 按意图加载 |
| 用户画像预热 | 画像向量 + 最近摘要 | Session + 5min | -- |
| GraphRAG 热点子图 | Top 30 路径 | 12h | 自用/送礼独立 |
| 配方骨架 | 情绪+场景+模式组合 | 1h | 自用/送礼独立 |
| 通感解码缓存 | 抽象表达->香调 | 24h | -- |

**缓存主动失效策略**:

| 变更事件 | 失效缓存 | 机制 |
|---------|---------|------|
| 知识图谱新增节点/边 | GraphRAG 热点子图 | 图谱写入时发布 Redis Pub/Sub -> 标记对应 key 过期 |
| 用户更新安全档案 | 外部偏好资源 | Session 内立即刷新（不等到 TTL） |
| 通感映射新增（人工审核后） | 通感解码缓存对应 key | 运营后台操作时主动删除 |
| 用户画像标签重大变更（>= 3 个标签同时变） | 用户画像预热缓存 | 异步任务完成后失效 |
| Layer 2 热检索新摘要写入 | 无（仅追加） | 摘要只追加不修改，无需失效 |

### 3.6 Setting 单例复用

> 详见产品需求文档 FR-6.5

全局配置（心理援助热线等）使用单例模式缓存，运营后台更新时主动刷新。

### 3.7 用户画像向量预加载

> 详见产品需求文档 FR-6.6

用户进入应用时预加载画像向量和最近摘要至 Redis，减少首次对话延迟。

### 3.8 外部偏好资源独立存储

> 详见产品需求文档 FR-6.7

安全档案、不喜欢成分、送礼历史等作为独立外部偏好资源存储，不嵌入用户画像主表。仅在会话意图 = `self_use` 时按需加载。

### 3.9 Layer 2 摘要记忆双写机制

> 详见产品需求文档 FR-6.8

- Redis + 外部存储双写。外部存储为权威源
- 压缩任务生命周期独立于 session 生命周期。session 关闭后压缩任务继续执行，结果通过 `session_id` 关联
- 外部存储先写 -> Redis 后写

### 3.10 全量会话原文存储

> 详见产品需求文档 FR-6.9

- 全量持久化 + 按月分区 + 索引优化
- 情绪标签从 metadata JSON 冗余存储为独立列 `emotion_tags`，支撑列表查询性能

### 3.11 游客临时数据存储

> 详见产品需求文档 FR-6.10

- `temp_conversations` 表，字段同 `conversations` + `browser_id`（设备指纹）+ `temp_session_token`
- 保留策略: 30 天。超期自动物理删除
- 注册成功后，根据 `browser_id` 将临时对话迁移至正式 `conversations` 表

### 3.12 Redis 配额计数器

> 详见业务流设计文档 §5.2

```
Redis Key 设计:
  quota:daily:{user_id}:sessions    -> INCR, TTL=次日00:00
  quota:daily:{user_id}:generations -> INCR, TTL=次日00:00
  quota:daily:{user_id}:deep_modes  -> INCR, TTL=次日00:00
  quota:monthly:{user_id}:cards     -> INCR, TTL=下月1日00:00
  quota:guest:{browser_id}:used     -> SET, TTL=30天
```

| 维度 | 计数点 | 存储 | 重置周期 |
|------|--------|------|---------|
| session_count | 新会话创建 | Redis INCR + PG 日志 | 每日 00:00 |
| gen_count | 推荐完成后 | Redis INCR + PG 日志 | 每日 00:00 |
| deep_count | 深度触发时 | Redis INCR + PG 日志 | 每日 00:00 |
| card_count | 提交卡片后 | PG 计数器 | 每月 1 日 |
| guest_used | 游客生成完 | browser_id + Redis | 30 天过期 |

**技术约束**:

| 约束 | 方案 |
|------|------|
| 计数延迟 | Redis INCR 原子操作，< 1ms |
| 并发安全 | Redis 单线程保证 INCR 不超卖 |
| 计数回滚 | 生成失败不扣减（先 INCR 后生成 -> 失败时 DECR） |
| 跨日重置 | Redis TTL = 次日 00:00 自动过期，无需定时任务 |
| 降级 | Redis 不可用 -> 跳过配额检查（不断用户路），标记 `quota_degraded` |

## §5 事件总线

> 详见业务流设计文档 §3.2

系统采用 **Redis Pub/Sub + 应用层 EventRouter** 的轻量事件总线。事件分为同步处理（阻塞当前请求）和异步处理（后台消费）。

### 5.1 命名规范

```
事件命名规范: {domain}.{action}.{status}
示例: emotion.classified.success
      crisis.detected.high
      session.completed
```

### 5.2 事件目录

| 事件 | 发布者 | 消费者 | 处理方式 | 延迟要求 |
|------|--------|--------|---------|---------|
| `emotion.classified` | M2 | M3（注入上下文） | 同步 | 内联延迟 |
| `crisis.detected` | M5 | M3（中断生成）+ M5（推送前端） | **同步+广播** | < 100ms |
| `intent.changed` | M5 | M3（重载上下文）+ M4（切换 Layer3） | 同步 | < 50ms |
| `gate.insufficient` | M5 | M3（阻断生成）+ M5（推送追问 UI） | 同步 | < 200ms |
| `generation.completed` | M3 | M7（记录反馈埋点）+ M4（摘要触发） | 异步 | < 5s |
| `session.completed` | M4 | M4（压缩任务）+ M1（标签更新） | 异步 | < 30s |
| `profile.updated` | M1 | M6（缓存失效）+ M3（预热画像） | 异步 | < 10s |
| `perfumer_queue.updated` | M3 | M5（推送通知） | 异步 | < 60s |
| `error.degraded` | 任一模块 | M8（监控告警） | 异步 | < 5s |
| `graph.edge_updated` | M6 | M3（缓存失效） | 同步 | < 1s |

### 5.3 事件结构规范

```
{
  "event_id": "evt_abc123",
  "event_type": "crisis.detected",
  "trace_id": "trace_xyz",        // 关联到请求入口
  "session_id": "sess_456",
  "timestamp": "ISO8601",
  "publisher": "M5",
  "payload": { ... },              // 事件特定数据
  "severity": "high" | "medium" | "low"
}
```

### 7.4 SSE 生成阶段状态协议

> 详见产品需求文档 FR-4.13

- 每次配方生成分配唯一的 `generation_id`
- 每个阶段开始时 SSE 推送 `{"phase": "skeleton"}` / `{"phase": "details"}` / `{"phase": "copy"}`
- 前端在 localStorage 记录 `{generation_id, last_phase}`
- 重连时前端发送 `resume_generation_id + resume_from_phase`，后端从对应阶段继续生成
- 如果 generation_id 已过期（生成完成 > 5min）-> 重新开始

## §8 并发与一致性

> 详见业务流设计文档 §3.4

| 场景 | 策略 | 细节 |
|------|------|------|
| **多标签页同用户** | 独立 session_id，Layer 2/3 共享 | 详见 FR-4.14。每个标签页独立 Layer 1 窗口。Layer 3 读取共享、写入乐观锁 |
| **压缩任务写摘要** | 外部存储先写 -> Redis 后写 | 详见 FR-6.8。压缩任务独立于 session 生命周期。session 已关闭仍可写入 |
| **用户同时精炼 + 发新消息** | FIFO 排队 + 合并检测 | 详见 FR-4.12。两条精炼指令不同维度/同方向 -> 合并为 1 次。反方向 -> 分别执行 |
| **画像异步更新 vs 实时读取** | 读取最新版本号，不阻塞 | M3 读取时用快照版本。M1 异步更新不影响当前生成。下次请求取新版本 |
| **冷归档查询与热检索并发** | 热检索先返回，冷归档异步追加 | M4 同时发起两条查询。热检索 < 20ms 即刻返回。冷归档下一轮注入 |
| **SSE 推送与前端状态** | generation_id + phase 协议 | 详见 FR-4.13。前端 localStorage 绑定 generation_id -> phase，断开后从断点续 |
| **Layer 1 溢出压缩** | 原子操作，压缩期间 Layer 1 只读 | 压缩任务获取写锁。若期间新消息到达 -> 先排队，压缩完成后写入 |

### 8.1 快速连续消息处理

> 详见产品需求文档 FR-4.12

- FIFO 排队 + 中间态提示 + 丢弃保护
- 合并条件收紧: 精炼指令合并仅当两条指令操作在**不同维度**或**同一维度同方向**时触发
  - 可合并: "再甜一点" + "清淡一点"（甜度增加 + 浓度降低 = 不同维度）
  - 不合并: "再甜一点" + "太甜了退回去"（甜度增加 + 甜度降低 = 同维度反方向）。分别执行，先增加后降低
- 合并上限: 2 条

### 8.2 多标签页并发

> 详见产品需求文档 FR-4.14

- 独立 session_id + Layer 2/3 共享 + 乐观锁 + 最后写入胜出

|------|----------|----------|

### 9.2 Layer 1 -- Agent 会话记忆

> 详见产品需求文档 FR-4.2

- **容量**: 1 轮首次对话（全量 Q&A）+ 5 轮最近对话（全量 Q&A）= 最多 6 轮
- **压缩规则**:
  - 触发条件：第 7 轮对话开始时（总轮次 > 6）
  - 压缩对象：第 2 轮至第 (N-1) 轮（即**不含首次**、**不含最后一轮**的所有中间轮次）
  - 压缩输出：
    - 中间轮次 <= 6 轮 -> 1 条结构化摘要
    - 中间轮次 >= 7 轮 -> 分批压缩为 **2 条摘要**（每条约 3-4 轮），保证每条摘要的信息密度合理
    - 写入 Layer 2 热检索层
  - 压缩后 Layer 1 保留：首次对话（全量）+ 最后一轮对话（全量）= 2 轮
  - **压缩比上限**: 单条摘要的输入上限为 6 轮对话。超过 6 轮中间对话时自动分批

### 9.3 Layer 2 -- 摘要记忆（热检索 + 冷归档）

> 详见产品需求文档 FR-4.3

**热检索层**:
- Redis: baseTTL + randomTTL 防雪崩。外部存储: 持久化双写
- 容量: 最近 N 轮（默认 N=10）

**冷归档层**:
- 存储: 对象存储（摘要原文）+ 轻量向量索引（FAISS，每日重建）
- 写入: 热检索层淘汰的摘要 -> 追加写入冷归档层
- 检索延迟: < 1s（批量索引，可接受低频查询的延迟）
- 查询触发: 热检索 topK 相似度均 < 0.5 阈值时，异步触发冷归档查询。结果在下一轮对话中作为补充上下文注入

### 9.4 Layer 3 -- 外部存储记忆（按意图加载）

> 详见产品需求文档 FR-4.4

- 安全档案、不喜欢成分、送礼历史、多样性滑块、场景习惯
- Layer 1+2 检索完成后 -> 意图门控决定加载

### 9.5 Session 生命周期管理

> 详见产品需求文档 FR-4.1

- session_id + 元数据 + 空闲超时 30min + 预加载基础画像
- session 结束时 LLM 异步生成摘要 -> 写入 Layer 2 热检索 + 记忆透明面板

### 9.6 会话总结与异步处理

> 详见产品需求文档 FR-4.5

- Session 结束 -> 异步生成摘要 -> Layer 2 双写
- 微会话合并（< 2min）: 时长 < 2min 的微会话原文全量保留，摘要层面合并到下一个长会话中

| 优化项 | 优化前延迟 | 优化后延迟 | 节省 | 覆盖场景 |
|--------|:---:|:---:|:---:|------|
| **BERT 替代 LLM 情绪分类** | LLM 1.5s | BERT 50ms | **-97% (1.45s)** | 70% 文本对话 |
| **Supervisor 去 LLM 化** | LLM 1.6s | 规则引擎 10ms | **-99% (1.6s)** | 100% 深度模式 |
| **Graph Explorer 纯检索** | Agent 模式 2.4s | 纯检索 0.5s | **-79% (1.9s)** | 100% 深度模式 |
| **LLM Composer 骨架先行** | 完整输出 2.2s | 仅骨架 0.7s | **-68% (1.5s)** | 100% 深度模式 |
| **Prompt Cache 预热** | 冷启动 3.2s | 热启动 1.5s | **-53% (1.7s)** | 老用户文本快速 |

### 11.5 Agent Gate 延迟影响

| 场景 | v2.5.1 骨架 | v2.6 骨架 | 增量 | 说明 |
|------|:---:|:---:|:---:|------|
| sufficient (95% 场景) | 不变 | **0ms** | **0ms** | Gate 跳过 LLM，判定在 < 1ms 完成 |
| partial + 工具补充成功 (3%) | 不变 | **+200ms** | +200ms | 调 1 次工具（查送礼历史/偏好），1 次轻量 LLM |

### 11.8 核心结论

1. **BERT 情绪分类是延迟优化的最大杠杆**：将 70% 文本对话的情绪识别从 LLM 1.5s 降至 BERT 50ms，单次对话节省 1.45s，加权平均收益显著
2. **Supervisor 去 LLM 化是关键架构决策**：触发条件已在 FR-3.12 中明确定义 5 种场景，每种场景的分派策略可通过策略矩阵硬编码，无需 LLM 动态判断
3. **v2.7 加权平均 ~1.37s 在 C 端可接受**：前提是情绪卡片路径占主流（67%）。如果文本路径占比超过 50%，平均将升至 ~1.6s（仍可接受，BERT 保障）
4. **深度模式 1.7s 骨架是云端模型物理极限**：即使经过全部优化，Supervisor+Subagentx2+Aggregator 的串行结构在云端模型上无法突破 1.5s。如需进一步下降，需要边缘部署（SDK/本地端侧推理）
5. **精炼对话 < 500ms 完全不受云端影响**：规则引擎 + 局部图谱操作，不涉及 LLM 推理。这是全系统延迟最稳的环节
6. **Agent Gate 的延迟代价极低（加权 ~10ms）**：95% 场景 Gate 判定 sufficient 后直接跳过（0ms）。partial 场景的 +200ms 工具补充改善了推荐质量。insufficient->追问 的首响应 ~0.6s 远快于勉强生成劣质推荐 -- 这是"延迟换质量"的正收益交换
7. **全链路 LLM 调用约束矩阵（§1.9）保障了最坏情况边界**：每层超时+重试+降级策略确保单次 LLM 调用失败不会拖垮整个链路的延迟。全链路累计 LLM 调用上限 10s（深度模式最坏情况 ~6.2s + 余量）
8. **Agent Gate 不是通用 Agent 循环，而是 2 个精确节点**：信息完整性 Gate 仅在推理前触发（~5% 场景触发 LLM），精炼语义 Gate 仅在第 2 次精炼后触发（~15% 精炼对话）。两个节点各 <=500ms 硬边界。在获得 Agent 灵活性的同时保持确定性编排的延迟可预测性
